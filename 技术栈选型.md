# 《山河策》技术栈选型

## 文档目的

本文档为《山河策》的开发提供技术栈选型依据，分析六套候选方案，给出推荐方案及完整技术栈，涵盖引擎、语言、地图、UI、数据、版本控制、构建打包、资产管线、测试和开发工具。

---

## 第一部分：候选方案对比

### 1. Godot Engine（GDScript / C#）

**概述：** 免费开源游戏引擎，场景-节点架构，Godot 4.x 的 2D 渲染、TileMap 和 UI 系统均为原生设计。

| 优势 | 劣势 |
|---|---|
| 2D 原生引擎，TileMap 原生支持六角网格 | GDScript 性能低于编译语言（回合制影响小） |
| Control 节点 UI 系统，适合密集面板 | C# 移动端导出稳定性待验证 |
| GDScript 语法接近 Python，学习成本低 | 策略游戏模板比 Unity 少 |
| 可视化编辑器，美术可独立预览布局 | 无 |
| 永久免费，无授权费和分成 | — |
| GodotSteam 插件成熟，Steam 集成完善 | — |
| 原生导出 Android/iOS | — |

**可行性：优秀** — 引擎特性与项目需求高度匹配。

---

### 2. Python + Pygame / Pygame-ce

**概述：** Python 低级 2D 游戏库，完全自主控制。

| 优势 | 劣势 |
|---|---|
| Python 易学易用，数据处理能力强 | 无内置 UI 框架，需从零搭建（4-6 周） |
| 完全控制，无引擎抽象 | 无内置 TileMap，六角网格需手动实现 |
| 数据驱动设计天然契合 | 无可视化编辑器 |
| — | 性能瓶颈：文化扩散、AI 计算 |
| — | 移动端导出极其困难 |
| — | Steam 集成需手动封装 C 库 |
| — | 无场景管理、动画、粒子系统 |

**可行性：差** — 从零搭建 UI 和地图系统与 20 周时间线不兼容。

---

### 3. Unity（C#）

**概述：** 行业标准游戏引擎，生态系统庞大，Asset Store 资源丰富。

| 优势 | 劣势 |
|---|---|
| 最大生态系统，策略游戏模板丰富 | 2D 是次要方向，TileMap 不如 Godot 直观 |
| C# 成熟，IDE 工具链完善 | 引擎笨重，2D 游戏包体偏大 |
| Steam 商业发布经验丰富 | 两套 UI 系统并存（uGUI / UI Toolkit），选择困难 |
| 原生移动端导出 | 2023 授权费争议影响信任 |
| — | 编辑器复杂，单人程序员学习成本高 |

**可行性：可行但非最优** — 能做，但为不需要的复杂度付费。

---

### 4. Web 方案（Phaser/PixiJS + React）

**概述：** Canvas/WebGL 游戏框架 + React UI，Electron/Tauri 打包桌面端。

| 优势 | 劣势 |
|---|---|
| JS/TS 广泛使用，React UI 开发效率高 | 双框架架构，Canvas 与 DOM 状态同步复杂 |
| 热重载开发体验好 | Electron 包体 100-200MB |
| 天然跨平台 | Steam 集成需额外封装 |
| — | 移动端非原生，性能和体验有妥协 |
| — | "网页套壳游戏"感知问题 |

**可行性：技术可行但架构别扭** — 双框架复杂度对地图密集型策略游戏不划算。

---

### 5. LibGDX（Java）

**概述：** 成熟的 Java 游戏框架，跨平台，Unciv（开源文明类游戏）基于此框架。

| 优势 | 劣势 |
|---|---|
| 4X 策略游戏有成功案例（Unciv） | Scene2D UI 过时且功能有限 |
| Java JIT 性能优于 Python/JS | Gradle 构建工具有学习曲线 |
| 跨平台（桌面 + Android） | 无可视化编辑器 |
| — | 社区规模小于 Godot/Unity |
| — | TileMap 支持不如 Godot 原生 |

**可行性：可行但老化** — 能做（Unciv 已证明），但 UI 系统是短板。

---

### 6. Bevy（Rust）

**概述：** 现代 ECS 架构游戏引擎，Rust 编写。

| 优势 | 劣势 |
|---|---|
| Rust 性能优秀，内存安全 | 尚未发布 1.0，API 不稳定 |
| ECS 架构适合复杂游戏状态 | 学习曲线陡峭（Rust 所有权 + ECS 思维） |
| — | 无可视化编辑器，UI 系统原始 |
| — | 生态系统小，教程和示例少 |
| — | 移动端导出实验性 |
| — | 无现成 Steam 集成插件 |

**可行性：不推荐** — 引擎未成熟，20 周硬截止日期下风险过高。

## 第三部分：推荐方案

### 推荐：Godot 4 + GDScript

**理由：**

1. **引擎匹配游戏** — 2D、基于网格、回合制、UI 密集，正是 Godot 的设计目标
2. **引擎匹配团队** — GDScript 低学习成本，可视化编辑器让美术可独立工作，单人程序员最大化生产力
3. **引擎匹配时间线** — TileMap + 基础回合循环可在数小时内搭建，把时间留给 AI、文化传播等核心系统
4. **移动端路径真实** — 原生导出 Android/iOS，回合制触控交互天然适配移动端
5. **永久零成本** — 无授权费、无分成
6. **Steam 集成成熟** — GodotSteam 插件覆盖 Steamworks SDK 全功能

## 第四部分：完整技术栈

### 4.1 游戏引擎

**Godot Engine 4.3+（最新稳定版）**

- 下载：[godotengine.org](https://godotengine.org)
- 使用标准版（非 .NET 版），GDScript 为主
- 如后期需要 C# 性能优化，可额外引入 .NET 版

### 4.2 编程语言

| 层级 | 语言 | 用途 |
|---|---|---|
| 主力 | GDScript | 全部游戏逻辑、UI、AI、数据加载 |
| 备选 | C# | 仅在性能分析确认瓶颈后使用（大地图寻路、文化扩散） |
| 扩展 | GDExtension（C++/Rust） | Phase 5 优化，非初始决策 |

**编码规范：** 尽量使用静态类型标注（`var health: int = 100`）以获得更好的 IDE 支持和错误检测。

### 4.3 地图与网格系统

- **TileMap / TileMapLayer**（Godot 4.3+）— 分层渲染：地形底层、地形覆盖、单位、迷雾、文化覆盖
- **网格类型** — 推荐**六角网格**，经典 4X 体验，邻居计算更自然
- **坐标系** — 轴坐标 (q, r)，使用 `local_to_map()` / `map_to_local()` 转换
- **自定义数据层** — 在 TileSet 上存储地形类型、移动力消耗、特殊标记（关隘、渡口）
- **寻路** — `AStarGrid2D`（内置，支持六角网格），自定义权重函数处理地形消耗
- **补给线检测** — BFS/泛洪填充，从部队到最近友方城市

### 4.4 UI 框架

- **Godot Control 节点**（内置）— 面板、菜单、滚动列表、工具提示、科技树图
- **关键节点** — PanelContainer、VBoxContainer、HBoxContainer、GridContainer、TabContainer、ScrollContainer、RichTextLabel
- **主题系统** — 项目级 Theme 资源，美术可独立迭代样式（字体、颜色、边框）
- **科技树可视化** — 自定义 Control 节点 + `_draw()` 绘制连线，或使用 GraphEdit

**字体支持：**
- 标题/书法字体：免费中文字体（如思源宋体）
- 正文/UI 字体：Noto Serif CJK 或思源宋体
- Godot 的 FontFile 资源支持完整 CJK 字符集

### 4.5 数据格式

| 用途 | 格式 | 说明 |
|---|---|---|
| 游戏数据表 | JSON | 单位属性、地形、科技树、建筑、事件、外交参数 |
| 引擎原生数据 | .tres（Godot Resource） | Theme、TileSet 配置、自定义 Resource 子类 |
| 存档 | JSON（优先）+ 二进制（可选） | JSON 便于调试和 bug 报告，二进制优化文件大小 |

**数据文件结构**（`data/` 目录）：
- `units.json` — 兵种属性
- `terrain.json` — 地形属性
- `tech_tree.json` — 科技树定义
- `events.json` — 随机事件模板
- `buildings.json` — 建筑属性
- `factions.json` — 七国特色
- `diplomacy.json` — 外交参数

**数据加载模式：**
```gdscript
class_name DataManager extends Node

var units: Dictionary = {}
var terrain: Dictionary = {}

func _ready() -> void:
    units = _load_json("res://data/units.json")
    terrain = _load_json("res://data/terrain.json")

func _load_json(path: String) -> Dictionary:
    var file = FileAccess.open(path, FileAccess.READ)
    var json = JSON.new()
    json.parse(file.get_as_text())
    return json.data
```

### 4.6 版本控制

- **Git**，托管于 GitHub（仓库已存在）
- **分支策略**：
  - `main` — 始终保持可运行
  - `dev` — 开发集成分支
  - `feature/*` — 功能分支（如 `feature/turn-system`、`feature/diplomacy-ui`）
- **.gitignore** — 排除 `.godot/` 缓存、导出构建、OS 特定文件
- **Git LFS** — 管理二进制资产（PNG、音频、字体）

### 4.7 构建与打包

- **Godot Export Templates** — 内置，一键导出 Windows / Linux / macOS / Android / iOS
- **Steam 集成** — [GodotSteam](https://github.com/CoaguCo-Industries/GodotSteam) 插件
  - Steam 云存档、成就、富状态、覆盖层
  - 在 Phase 5 集成，游戏功能完成后再接入
- **CI/CD**（可选）— GitHub Actions 自动构建、运行测试、生成可下载构建

### 4.8 资产管线

| 类型 | 工具 | 说明 |
|---|---|---|
| 像素美术（32x32） | Aseprite（$20）或 LibreSprite（免费） | 导出 PNG 精灵表，Godot 中设置纹理过滤为 Nearest |

**TileSet 创建流程：** 精灵表导入 Godot → TileSet 编辑器定义碰撞形状和自定义数据层 → TileMap 使用

### 4.9 测试方案

### 5.2 Autoload 单例（全局管理器）

| 单例 | 职责 |
|---|---|
| `GameManager` | 回合循环、当前阶段追踪、游戏开始/结束 |
| `DataManager` | 加载和提供所有 JSON 数据表访问 |
| `EventManager` | 随机事件队列管理、条件触发 |
| `SignalBus` | 纯信号发射节点，系统间解耦通信 |

### 5.3 信号驱动架构

回合结束信号流示例：

```
GameManager.end_turn()
  → 发射 "turn_ending" 信号
  → ResourceManager.collect_resources()
  → CultureSystem.spread_culture()
  → MoraleSystem.update_morale()
  → EventSystem.check_and_trigger_events()
  → AISystem.execute_ai_turns()
  → SeasonSystem.advance_season()
  → 发射 "turn_started" 信号
  → UI 更新
```

各系统独立开发和测试，小团队并行工作的关键。

### 5.4 数据驱动设计原则

> 策划必须在阶段 0 冻结数据格式（JSON/CSV 结构），程序严禁硬编码数值。— 策划案

**实施模式：**

1. 策划定义数据 schema（JSON 字段名和类型）
2. 程序构建数据加载和运行时查询系统
3. 所有平衡数值（单位属性、地形消耗、科技效果、外交修正、事件权重）存放在 JSON 文件中
4. 程序不写 `if faction == "Qin": attack += 20`，而是 `attack += DataManager.get_faction_bonus(faction_id, "attack")`

## 第六部分：风险与对策

| 风险 | 对策 |
|---|---|
| GDScript 大地图性能瓶颈 | 先分析后优化，仅对确认的热点使用 GDExtension（C++） |
| Godot C# 移动端导出不稳定 | 以 GDScript 为主力语言，完全避免 C# 移动端路径 |
| 单人程序员成为瓶颈 | 数据驱动设计让策划可独立调数值，美术可在编辑器中独立调整 UI 和 TileSet |
| Godot 版本更新导致不稳定 | 锁定特定 Godot 次版本（如 4.3.x），不在项目中途升级 |
| Steam 集成复杂 | GodotSteam 文档完善、社区广泛使用，在 Phase 5 功能完成后再集成 |
| 范围蔓延（策划案对 20 周来说过于宏大） | 严格遵循分阶段交付，落后时从当前阶段砍范围，不推迟到后续阶段 |

## 第七部分：与开发计划对齐

### 阶段 0（第 1~2 周）— 地基搭建

- 搭建 Godot 项目、Git 仓库、项目目录结构
- 实现六角 TileMap + 地形自定义数据层
- 定义 JSON 数据 schema（单位、地形、城市）
- 构建 `canMove` 移动检测（含地形消耗计算）
- **交付**：控制台验证的六角网格移动逻辑

### 阶段 1（第 3~6 周）— 最小可玩原型

- 回合循环（GameManager Autoload）
- 资源系统（粮、钱、铁）
- 基础战斗结算
- 简易 AI（随机移动/攻击）
- 最小 UI（资源栏、单位选择、移动/攻击指令）
- **交付**：可运行 .exe，一城一玩家对一城一 AI

### 阶段 2（第 7~9 周）— 地理与外交

- 不可通行地形（山脉、无渡口河流）
- 关隘特殊战斗规则
- 外交 UI 面板 + 好感度系统
- AI 外交决策逻辑
- **交付**：玩家需绕路而行，可与 AI 进行基本政治交易

### 阶段 3（第 10~12 周）— 学派与文化传播

- 学派选择 UI + Buff 管理系统
- 文化扩散算法（城市泛洪填充）
- 地图文化覆盖层（瓦片着色）
- **交付**：玩家可选择治国路线，文化渗透导致敌城叛乱

### 阶段 4（第 13~16 周）— 战国特色

- 七国特色兵种和国家加成
- 随机事件引擎（JSON 驱动事件模板）
- 半文言文本集成
- **交付**：完整游戏体验，各国玩法迥异

### 阶段 5（第 17~20 周）— 打磨与发布

- 存档/读档（JSON 序列化）
- 新手引导
- UI 美化（主题、动画、过渡）
- Steam 集成（GodotSteam）
- 音频集成
- **交付**：Steam / itch.io 1.0 版本

## 结语

> 先跑起来，再谈艺术。Godot 4 + GDScript 是当前约束条件下的最优选择——2D 原生、零成本、低门槛、移动端路径清晰。三人同心，二十周可期。
