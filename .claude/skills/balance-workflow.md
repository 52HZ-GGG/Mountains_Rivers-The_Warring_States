# 数值平衡性调整工作流

> 本文档定义《山河策》数值平衡性调整的标准流程。
> **核心原则：机制以 `docs/机制概览/` 文档为准，不以程序实现为准。**

---

## 0. 设计原则

### 0.1 兵种性价比设计原则

**核心理念**：性价比天然不平衡，但不可相差太大让某些兵种具有无脑优势。

| 原则 | 说明 |
|:---|:---|
| **低级兵种性价比高** | T1（民兵）性价比最高，T2-T5 逐级递减 |
| **高级兵种性价比低** | 通过征兵人口限制、特殊资源需求等因素平衡 |
| **差距可控** | 相邻等级性价比差距不超过 30%，避免无脑选择 |
| **定位差异化** | 同等级兵种应有明确的定位差异，而非单纯数值碾压 |

### 0.2 性价比等级划分

| 等级 | 兵种类型 | 性价比范围 | 设计意图 |
|:---:|:---|:---:|:---|
| T1 | 廉价炮灰（民兵） | 1.3-1.5 | 性价比最高，但战斗力弱 |
| T2 | 标准兵种（步兵/弓兵/战车等） | 0.6-1.0 | 主力兵种，性价比适中 |
| T3 | 精锐兵种（铁甲兵/护卫骑兵等） | 0.4-0.6 | 高战斗力，性价比下降 |
| T4 | 高级兵种（重骑兵/大翼等） | 0.3-0.4 | 顶级战斗力，性价比低 |
| T5 | 顶级兵种（楼船等） | 0.2-0.3 | 特殊定位，性价比最低 |

### 0.3 建筑性价比设计原则

**核心理念**：与兵种类似，建筑也遵循"高级低性价比"原则，但差距需控制在合理范围内。

| 原则 | 说明 |
|:---|:---|
| **基础建筑性价比高** | 经济建筑（农田/市集/伐木场等）是发展的基石，性价比应最高 |
| **高级建筑性价比低** | 学派建筑/特殊建筑（王宫等）效果强大但造价昂贵，性价比应较低 |
| **差距可控** | 同类别内最高与最低性价比差距不超过 50%，避免"永不建"选项 |
| **维护费合理** | 维护费应与收益匹配，避免"建了反而亏钱"的情况 |
| **定位差异化** | 同类别建筑应有明确的定位差异（如祠堂专注安定、学宫兼顾文化） |

**建筑性价比层级**：

| 层级 | 建筑类型 | ROI目标 | 设计意图 |
|:---:|:---|:---:|:---|
| 基础 | 经济建筑（农田/市集/伐木场等） | 8-15回合 | 性价比最高，发展基石 |
| 标准 | 军事/防御/政治建筑 | 12-20回合 | 性价比适中，战略支撑 |
| 高级 | 学派建筑/特殊建筑 | 15-25回合 | 效果强大，性价比下降 |
| 顶级 | 王宫/奇观 | 20-30回合 | 终极建筑，性价比最低 |

**关键约束**：
- 经济建筑ROI不应低于 5 回合（过快会导致无脑建造）
- 学派建筑ROI不应超过 30 回合（过慢会导致无人问津）
- 维护费占收益比例不应超过 50%（避免"建了反而亏钱"）

### 0.4 攻城器械设计原则

- **基础属性低**：攻城器械的主要价值是 siege_bonus ×3.0，基础属性仅需自保
- **特殊资源需求**：工匠/精铁需求体现"精密器械"的定位
- **性价比可接受**：攻城器械的性价比应在 T3-T4 水平（0.3-0.5）

### 0.5 特殊资源平衡原则

- **产出周期合理**：高阶兵种的特殊资源获取周期应控制在 2-3 回合
- **稀缺但不可缺**：特殊资源应稀缺但不至于让兵种"造不出来"
- **调整方向**：优先调整产出量，而非降低兵种需求

### 0.6 科技性价比设计原则

**核心理念**：强力科技性价比低，精通升级性价比低，避免"无脑研究最强科技"。

| 原则 | 说明 |
|:---|:---|
| **强力科技性价比低** | 效果越强的科技，每金币/回合获得的效果增量应越低 |
| **精通性价比低** | 精通升级的边际收益应递减，总投入产出比低于基础研究 |
| **时代递进** | 同类别内，晚期科技的性价比应低于早期科技 |
| **前置成本纳入** | 计算性价比时需纳入前置科技的总成本（金币+回合） |
| **特殊条件加价** | 有特殊条件（控制城市/声望/奇观）的科技，因获取难度高，性价比可略高 |

**科技性价比公式**：

```
总效果值 = Σ(正面效果值) - Σ(|负面效果值|)
总成本 = 金币成本 + 研究回合数 × 每回合机会成本
性价比 = 总效果值 / 总成本
前置成本 = Σ(前置科技的金币成本 + 回合成本)
全链路性价比 = 总效果值 / (总成本 + 前置成本)
```

**性价比层级**：

| 层级 | 科技类型 | 全链路性价比范围 | 设计意图 |
|:---:|:---|:---:|:---|
| 基础 | 无前置的早期科技 | 0.15-0.25 | 入门科技，性价比最高 |
| 标准 | 有1-2个前置的中期科技 | 0.08-0.15 | 主力科技，性价比适中 |
| 高级 | 长前置链的晚期科技 | 0.04-0.08 | 强力科技，性价比低 |
| 终极 | 需多前置+特殊条件的终极科技 | 0.02-0.04 | 终极科技，性价比最低 |
| 精通 | 任何科技的精通升级 | 0.01-0.03 | 边际收益递减 |

**精通升级特殊约束**：
- 精通每级增量应递减或持平，不可递增
- 精通总投入（金币+帛书）的性价比应低于基础研究
- 精通 5 级满级的总效果不应超过基础效果的 100%

## 1. 流程总览

```
读取机制文档 → 读取 JSON 数据 → 分析合理性 → 提出异常 → 确认修改 → 四步更新
```

## 2. 第一步：读取机制文档（源 of Truth）

根据要调整的系统，读取对应的机制文档。`docs/机制概览/` 下每个文件对应一个系统：

| 系统 | 文档 | 关联 JSON |
|:---|:---|:---|
| 战斗/攻城 | `战斗系统.md` | `units.json`、`balance_params.json`、`terrain.json` |
| 城市/防御建筑 | `城市经营系统.md` | `cities.json`、`buildings.json`、`balance_params.json` |
| 科技 | `科技系统.md` | `tech_tree.json`、`tech_events.json` |
| 外交 | `外交系统.md` | `diplomacy.json` |
| 经营/资源 | `经营系统.md` | `balance_params.json`、`buildings.json` |
| 粮食/人口/征兵 | `粮食人口征兵系统.md` | `balance_params.json` |
| 学派 | `学派系统.md` | `schools.json` |
| 民心/稳定 | `民心与稳定性系统.md` | `balance_params.json` |
| 季节 | `季节系统.md` | `balance_params.json` |
| 事件 | `随机事件系统.md` | `events.json` |
| 文化 | `文化系统.md` | `balance_params.json` |
| 大夫 | `大夫系统.md` | `ministers.json` |
| 地理/天堑 | `地理与天堑系统.md` | `terrain.json`、`big_map_terrain.json` |

**通用读取顺序**：
1. 对应系统的机制文档（定义规则）
2. `docs/机制概览/游戏机制总览.md`（系统交互关系，需要时查阅）
3. `docs/策划决策/` 相关决策记录（历史调整依据）
4. `docs/机制概览/数值平衡参考手册.md`（跨系统的数值基准，需要时查阅）

**关键规则**：
- 如果文档与代码冲突，以文档为准
- 如果发现代码未实现文档定义的机制，记录为 issue 而非修改文档
- 如果文档之间冲突，以具体系统的专项文档为准（如战斗以 `战斗系统.md` 为准）

## 3. 第二步：读取 JSON 数据

根据要调整的系统，读取对应的 JSON 文件（见上表"关联 JSON"列）。所有 JSON 位于 `data/` 目录下。

**常用 JSON 速查**：

| 文件 | 内容 |
|:---|:---|
| `units.json` | 兵种攻/防/HP/速度/射程/特殊技能/势力变体 |
| `balance_params.json` | 全局平衡参数（城市等级/攻城/士气/伤害公式/资源/民心等） |
| `buildings.json` | 建筑定义（成本/效果/前置条件） |
| `cities.json` | 50 城定义（坐标/辖区/初始建筑） |
| `factions.json` | 7 势力定义（AI 性格/加成/特色兵种） |
| `terrain.json` | 11 种地形（移动力/防御/攻击修正） |
| `tech_tree.json` | 54 项科技（前置/成本/效果） |
| `diplomacy.json` | 外交参数（阈值/衰减/AI 决策） |
| `events.json` | 随机事件（条件/选项/效果/冷却） |
| `schools.json` | 诸子百家定义 |
| `ministers.json` | 官员定义 |

## 4. 第三步：分析合理性

根据调整的系统类型，选择对应的分析维度。

### 4.1 战斗/兵种类

**核心公式**（乘法模型，COEFF=20）：
```
伤害 = 攻击力 × 攻击buff × 克制系数 × 20 / (20 + 防御力 × 防御buff)
回合数 = ceil(HP / 伤害)
```

**设计基准**：
- 同级兵种裸数值互打：**5-6 回合**
- 克制关系下的优势方：**3-4 回合**
- 单个步兵攻城：**极慢（30+ 回合）**

**检查项**：
- 同类互打回合数（< 4 或 > 8 → 异常）
- 克制关系伤害倍率（< 1.15 或 > 1.5 → 异常）
- 攻城器械 vs 普通单位差异（< 1.5 倍 → 攻城器械缺乏存在感）
- 城防值减伤率（> 50% → "打不动"）
- 远程 vs 近战反击（近战反击 > 远程攻击 → 不合理）

### 4.2 经济/资源类

**检查项**：
- 产出/消耗比：资源产出能否支撑基本运营
- 投资回报期：建筑/科技的回本周期是否合理（过短→无意义，过长→没人建）
- 边际收益递减：高级选项的性价比应低于基础选项

### 4.3 城市/防御类

**检查项**：
- 各等级城市的攻防节奏（村快破、都不易攻）
- 防御建筑的战术价值（无此建筑 vs 有此建筑的差异）
- 驻军对防御的影响

### 4.4 通用数值关系

- **成本/收益比**：高成本选项应有对应高收益，但不应线性增长
- **梯度差异**：同系列数值（如 Lv1→Lv5）应有清晰递增感
- **阈值合理性**：触发条件的阈值不应过高（永远触发不了）或过低（总是触发）

## 5. 第四步：提出异常

发现异常时，输出格式：

```markdown
### 异常：[简短描述]

**当前数值**：[具体值]
**问题**：[为什么不合理]
**建议调整**：[具体数值建议]
**影响范围**：[哪些兵种/系统受影响]
```

**必须等用户确认后才执行修改。**

---

## 6. 第五步：确认修改后的四步更新

用户确认修改后，按以下顺序更新：

### 6.1 更新 JSON 数据文件

修改对应的 `data/*.json` 文件。注意：
- 修改后必须验证 JSON 格式正确（`python3 -c "import json; json.load(open('文件路径'))"`）
- 字段名不可更改（Schema 冻结），只改值

### 6.2 更新决策记录

在 `docs/策划决策/` 下追加或更新决策记录，包含：
- 调整背景与原因
- 调整前后的数值对比表
- 伤害验证表（关键对战场景的回合数）
- 程序变更清单（如需改代码）

### 6.3 更新机制概览文档

更新 `docs/机制概览/` 下**所有受影响的文件**。原则：改了哪个系统的数值，就更新对应的机制文档。

| 改动内容 | 需要更新的文档 |
|:---|:---|
| 兵种数值 | `战斗系统.md` 兵种表 + `数值平衡参考手册.md` 兵种表 |
| 战斗公式/参数 | `战斗系统.md` 对应章节 + `数值平衡参考手册.md` 战斗推算 |
| 城市/防御建筑 | `城市经营系统.md` 城市等级表 + 防御建筑表 |
| 科技效果/成本 | `科技系统.md` 相关条目 |
| 外交参数 | `外交系统.md` 相关章节 |
| 经济/建筑 | `经营系统.md` + `城市经营系统.md` 建筑表 |
| 势力加成 | `战斗系统.md` 势力变体表 + `游戏机制总览.md` |
| 民心/稳定参数 | `民心与稳定性系统.md` 相关章节 |

**注意**：一个数值可能影响多个文档。例如改兵种攻击 → 同时更新战斗系统和数值平衡参考手册。

### 6.4 更新数值平衡参考手册

`docs/机制概览/数值平衡参考手册.md` 需要同步更新：
- §2.1 兵种基础数值总览
- §2.4 战斗推算（同类对战、克制对战、攻城推算）
- §3.2 城池数值与攻城回合
- §3.3 防御建筑投资回报
- §5 异常值检测（如有）

## 7. 额外检查项

| 检查项 | 说明 |
|:---|:---|
| GUT 测试 | 如果改了公式，检查 `tests/unit/` 下相关测试是否需要更新预期值 |
| CLAUDE.md | 如果改了核心数值，更新 `CLAUDE.md` 中的快速定位指南 |
| 势力变体 | 改基础兵种时，检查 `faction_variants` 是否需要同步调整 |
| 决策记录一致性 | 确保新决策记录与历史决策记录不矛盾 |

---

## 8. 禁止事项

- ❌ 不修改 JSON Schema（字段名）
- ❌ 不在代码中硬编码数值（必须从 JSON 读取）
- ❌ 不在未确认的情况下修改数值
- ❌ 不跳过决策记录直接改数据
- ❌ 不只改 JSON 不更新文档
