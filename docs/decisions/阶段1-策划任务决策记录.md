## 关键决策

### 决策 25：建筑系统采用「5种基础建筑 + 线性升级」模型

**选定方案**：
- 5种建筑：农田（粮+50/级）、市集（金+40/级）、矿场（铁+20/级）、兵营（征兵效率+25%/级）、城墙（城防+15%/级）
- 经济建筑最高3级，军事建筑最高2级
- 升级成本 = 基础成本 × 1.5^当前等级

**理由**：
- MVP阶段不做复杂建筑树，5种建筑覆盖经济和军事两大方向
- 线性叠加简单直观，便于策划和玩家理解
- 兵营和城墙成本高且需铁矿，体现"耕战"思想
**已落地**：`data/buildings.json`、`docs/数值公式.md` §9

---

### 决策 26：征兵消耗采用「即时资源 + 兵营折扣」模式

**选定方案**：
- 征兵即时消耗 gold/food/iron（来自 units.json 的 cost 字段）
- 兵营每级减 5% 金钱消耗
- 征兵时间默认1回合，兵营加速但保底1回合

**理由**：
- 阶段1不做征兵队列UI，简化为即时消耗
- 兵营折扣提供明确的升级动力
- 1回合保底避免"秒出兵"破坏节奏
**已落地**：`docs/数值公式.md` §10、`data/balance_params.json` recruitment

---

### 决策 27：事件触发时机限定为「回合结算时」

**选定方案**：
- 事件仅在 `turn_start` 时机触发（玩家回合开始时）
- 不实现战斗中事件、移动中事件等

**理由**：
- 简化EventManager实现，阶段1只需一个触发入口
- 回合开始时触发让玩家在行动前做出决策
- 阶段4可扩展 `turn_end`、`on_battle`、`on_move` 等时机
**已落地**：`scripts/autoload/event_manager.gd`、`data/events.json` trigger.type

---

### 决策 28：事件冷却机制为「同一事件间隔3回合」

**选定方案**：
- 每个事件触发后进入3回合冷却
- 冷却期间该事件不会被判定触发
- 冷却在 `turn_ended` 信号时递减

**理由**：
- 防止同一事件反复刷屏，保持新鲜感
- 3回合约等于1个季节周期，间隔合理
- 冷却值可通过 `balance_params.json` 的 `event_cooldown_turns` 调整
**已落地**：`scripts/autoload/event_manager.gd`、`data/balance_params.json` event_cooldown_turns

---

### 决策 29：公式常量集中管理为 balance_params.json

**选定方案**：
- 将数值公式.md中散布的所有常量抽取到 `data/balance_params.json`
- 程序通过 `DataManager.get_balance_param("path.to.param")` 点分路径访问
- 包含：combat、counter_matrix、resources、morale、supply、recruitment、season_cycle、culture、general

**理由**：
- 数据驱动铁律的彻底落地：连常量也不硬编码
- 单一数据源，修改常量无需改代码
- 点分路径访问保持接口简洁
**已落地**：`data/balance_params.json`、`scripts/autoload/data_manager.gd`

---

### 决策 30：7×7克制矩阵数据化存入 balance_params.json

**选定方案**：
- `counter_matrix` 作为嵌套字典存入 `balance_params.json`
- 程序通过 `DataManager.get_counter_multiplier(attacker_id, defender_id)` 访问
- 矩阵值与阶段0决策17的克制表完全一致

**理由**：
- 克制系数是最常调整的平衡参数，数据化后可热更新
- 阶段4平衡测试时策划可直接修改JSON而不需程序配合
**已落地**：`data/balance_params.json` counter_matrix

---

### 决策 31：阶段1落地15个事件覆盖4种类型

**选定方案**：
- 经济事件4个：五谷丰登、商旅云集、旱魃为虐、矿脉发现
- 军事事件4个：匪寇蜂起、逃兵四起、敌军细作、练兵有方
- 民心事件4个：祭祀大典、瘟疫流行、贤士来投、民怨沸腾
- 季节事件3个：春耕农忙（春）、寒冬苦厄（冬）、丰年有余（秋）

**理由**：
- 15个事件对MVP足够，每2~3回合触发一次（概率10%~15%）
- 4种类型覆盖经济/军事/民心/季节，不做外交和文化事件
- 季节事件概率1.0（每到季节必触发），提供稳定的季节感
- 阶段3/4再扩展到30+
**已落地**：`data/events.json`

### 决策 32：事件同时支持「无选项自动」和「有选项A/B」两种结构

**选定方案**：
- `effects` 和 `options` 互斥：一个事件只能有其中一种
- 无选项事件：`effects` 非null、`options` 为null，效果直接生效
- 有选项事件：`effects` 为null、`options` 非null，发射 `event_triggered` 信号等待UI处理
- 玩家选择后调用 `EventManager.resolve_event_choice()` 应用效果

**理由**：
- 无选项事件（如五谷丰登）提供被动随机性
- 有选项事件（如旱魃为虐）增加玩家决策感
- 互斥结构简化触发逻辑，避免混合处理
- 15个事件中9个有选项（60%），符合决策感预期
**已落地**：`data/events.json`、`scripts/autoload/event_manager.gd`

---

### 决策 33：政治建筑全国限建 2 座

**选定方案**：
- 祠堂、学宫、驿站、粮仓为政治建筑（category: "political"）
- 每种全国限建 2 座，通过 `max_national_count` 字段控制
- 不同城市可各建 1 座，但同种建筑全国不超过上限

**理由**：
- 限建防止玩家堆叠同类建筑获取过高收益
- 全国限 2 座比每城限 1 座更灵活，允许玩家在两个关键城市布局
- 通过 JSON 字段控制而非硬编码，便于后续调整
**已落地**：`data/buildings.json`

---

### 决策 34：奇观独立为 wonders.json

**选定方案**：
- 奇观使用独立数据文件 `wonders.json`，与建筑系统分离
- 全局唯一：每个奇观全世界只能存在一座
- 不占建筑槽位，效果覆盖全国
- 不可升级，效果固定

**理由**：
- 奇观与普通建筑的核心差异（全局唯一 vs 可重复、全国效果 vs 单城效果）决定了需要独立数据结构
- 独立文件便于策划单独调整奇观数值，不影响建筑平衡
- 奇观建造成本远高于建筑（500~1000金 vs 100~400金），体现"千年大计"的定位
**已落地**：`data/wonders.json`、`scripts/autoload/data_manager.gd`

---

### 决策 35：王宫为可选建造的特殊建筑

**选定方案**：
- 王宫为特殊建筑（category: "special"），每国限建 1 座
- 非首都默认建筑，需要玩家主动花费 400金+100铁 建造
- 3 级可升级，提供民心+税收+外交声望的综合加成

**理由**：
- 可选建造增加玩家决策点：是先建王宫还是先建经济/军事建筑
- 高成本体现"营建宫室"的重量感
- 综合加成（民心+5/级、税收+10%/级、声望+5/级）使其成为中期战略目标
- 阶段3+增加首都前置条件
**已落地**：`data/buildings.json`

## 新增 Schema 定义

### buildings.json

| 字段 | 类型 | 说明 |
|:---|:---|:---|
| id | String | 建筑唯一ID |
| name | String | 中文名 |
| description | String | 描述 |
| category | String | 分类：economic / military / political / special |
| cost_gold | int | 建造金钱成本 |
| cost_iron | int | 建造铁矿成本 |
| build_turns | int | 建造所需回合数 |
| max_level | int | 最高等级 |
| upkeep_gold | int | 每级每回合维护费 |
| effects | Dictionary | 建筑效果（键为效果类型，值为每级数值） |
| upgrade_cost_multiplier | float | 升级成本倍率 |
| max_national_count | int/null | 全国限建数量（null=不限） |

### events.json

| 字段 | 类型 | 说明 |
|:---|:---|:---|
| id | String | 事件唯一ID |
| title | String | 四字文言标题 |
| description | String | 半文言描述（30字内） |
| category | String | 分类：economy / military / morale / season |
| trigger.type | String | 触发时机：turn_start |
| trigger.probability | float | 基础触发概率 0.0~1.0 |
| trigger.conditions | Dictionary | 触发前置条件（season/morale_min/morale_max/turn_min/turn_max） |
| effects | Dictionary/null | 无选项事件直接效果 |
| options | Array/null | 有选项事件的选择项 |

### balance_params.json

顶层 sections：`combat, counter_matrix, resources, morale, supply, recruitment, season_cycle, culture, general, event_cooldown_turns`

### wonders.json

| 字段 | 类型 | 说明 |
|:---|:---|:---|
| id | String | 奇观唯一ID |
| name | String | 中文名 |
| description | String | 半文言描述 |
| cost_gold | int | 建造金钱成本 |
| cost_iron | int | 建造铁矿成本 |
| build_turns | int | 建造所需回合数 |
| upkeep_gold | int | 每回合维护费 |
| effects | Dictionary | 全国效果（`*_national` 后缀的 key） |
| flavor_text | String | 历史/文化引用 |

## 程序接口约定

EventManager 需要 GameManager 提供以下资源管理接口（已实现）：

| 方法 | 说明 |
|:---|:---|
| `get_player_morale() -> int` | 获取玩家当前民心 |
| `get_player_food/gold/iron() -> int` | 获取玩家当前资源 |
| `apply_food/gold/iron/morale/population/troops_delta(int)` | 应用资源变化 |

DataManager 新增接口：

| 方法 | 说明 |
|:---|:---|
| `get_all_events() -> Array` | 获取所有事件定义 |
| `get_event(id) -> Dictionary` | 按ID获取事件 |
| `get_building(id) -> Dictionary` | 按ID获取建筑定义 |
| `get_all_buildings() -> Array` | 获取所有建筑定义 |
| `get_buildings_by_category(cat) -> Array` | 按分类获取建筑 |
| `get_all_wonders() -> Array` | 获取所有奇观定义 |
| `get_wonder(id) -> Dictionary` | 按ID获取奇观 |
| `get_balance_param(path) -> Variant` | 点分路径访问平衡参数 |
| `get_counter_multiplier(atk, def) -> float` | 获取兵种克制倍率 |
| `get_current_season(turn) -> String` | 根据回合数获取当前季节 |
