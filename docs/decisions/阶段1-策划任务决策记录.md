## 关键决策

### 决策 25：建筑系统采用「5种基础建筑 + 线性升级」模型

**选定方案**：
- 5种建筑：农田（粮+50/级）、市集（金+40/级）、矿场（铁+20/级）、兵营（征兵效率+25%/级）、城墙（城防+15%/级）
- 经济建筑最高3级，军事建筑最高2级
- 升级成本 = 基础成本 × 1.5^当前等级

**理由**：
- MVP阶段不做复杂建筑树，5种建筑覆盖经济和军事两大方向
- 线性叠加简单直观，便于策划和玩家理解
- 兵营和城墙成本高且需铁矿，体现"耕战"思想
**已落地**：`data/buildings.json`、`docs/数值公式.md` §9

### 决策 26：征兵消耗采用「即时资源 + 兵营折扣」模式

**选定方案**：
- 征兵即时消耗 gold/food/iron（来自 units.json 的 cost 字段）
- 兵营每级减 5% 金钱消耗
- 征兵时间默认1回合，兵营加速但保底1回合

**理由**：
- 阶段1不做征兵队列UI，简化为即时消耗
- 兵营折扣提供明确的升级动力
- 1回合保底避免"秒出兵"破坏节奏
**已落地**：`docs/数值公式.md` §10、`data/balance_params.json` recruitment

---

### 决策 27：事件触发时机限定为「回合结算时」

**选定方案**：
- 事件仅在 `turn_start` 时机触发（玩家回合开始时）
- 不实现战斗中事件、移动中事件等

**理由**：
- 简化EventManager实现，阶段1只需一个触发入口
- 回合开始时触发让玩家在行动前做出决策
- 阶段4可扩展 `turn_end`、`on_battle`、`on_move` 等时机
**已落地**：`scripts/autoload/event_manager.gd`、`data/events.json` trigger.type

---

### 决策 28：事件冷却机制为「同一事件间隔3回合」

**选定方案**：
- 每个事件触发后进入3回合冷却
- 冷却期间该事件不会被判定触发
- 冷却在 `turn_ended` 信号时递减

**理由**：
- 防止同一事件反复刷屏，保持新鲜感
- 3回合约等于1个季节周期，间隔合理
- 冷却值可通过 `balance_params.json` 的 `event_cooldown_turns` 调整
**已落地**：`scripts/autoload/event_manager.gd`、`data/balance_params.json` event_cooldown_turns

**选定方案**：

- 点分路径访问保持接口简洁
**已落地**：`data/balance_params.json`、`scripts/autoload/data_manager.gd`

### 决策 31：阶段1落地15个事件覆盖4种类型

**选定方案**：
- 经济事件4个：五谷丰登、商旅云集、旱魃为虐、矿脉发现
- 军事事件4个：匪寇蜂起、逃兵四起、敌军细作、练兵有方
- 民心事件4个：祭祀大典、瘟疫流行、贤士来投、民怨沸腾
- 季节事件3个：春耕农忙（春）、寒冬苦厄（冬）、丰年有余（秋）

**理由**：
- 15个事件对MVP足够，每2~3回合触发一次（概率10%~15%）
- 4种类型覆盖经济/军事/民心/季节，不做外交和文化事件
- 季节事件概率1.0（每到季节必触发），提供稳定的季节感
- 阶段3/4再扩展到30+
**已落地**：`data/events.json`

### 决策 32：事件同时支持「无选项自动」和「有选项A/B」两种结构

**选定方案**：
- `effects` 和 `options` 互斥：一个事件只能有其中一种
- 无选项事件：`effects` 非null、`options` 为null，效果直接生效
- 有选项事件：`effects` 为null、`options` 非null，发射 `event_triggered` 信号等待UI处理
- 玩家选择后调用 `EventManager.resolve_event_choice()` 应用效果

**理由**：
- 无选项事件（如五谷丰登）提供被动随机性
- 有选项事件（如旱魃为虐）增加玩家决策感
- 互斥结构简化触发逻辑，避免混合处理
- 15个事件中9个有选项（60%），符合决策感预期
**已落地**：`data/events.json`、`scripts/autoload/event_manager.gd`

### 决策 33：政治建筑全国限建 2 座

**选定方案**：
- 祠堂、学宫、驿站、粮仓为政治建筑（category: "political"）
- 每种全国限建 2 座，通过 `max_national_count` 字段控制
- 不同城市可各建 1 座，但同种建筑全国不超过上限

**理由**：
- 限建防止玩家堆叠同类建筑获取过高收益
- 全国限 2 座比每城限 1 座更灵活，允许玩家在两个关键城市布局
- 通过 JSON 字段控制而非硬编码，便于后续调整
**已落地**：`data/buildings.json`

### 决策 34：奇观独立为 wonders.json

**选定方案**：
- 奇观使用独立数据文件 `wonders.json`，与建筑系统分离
- 全局唯一：每个奇观全世界只能存在一座
- 不占建筑槽位，效果覆盖全国
- 不可升级，效果固定

**理由**：
- 奇观与普通建筑的核心差异（全局唯一 vs 可重复、全国效果 vs 单城效果）决定了需要独立数据结构
- 独立文件便于策划单独调整奇观数值，不影响建筑平衡
- 奇观建造成本远高于建筑（500~1000金 vs 100~400金），体现"千年大计"的定位
**已落地**：`data/wonders.json`、`scripts/autoload/data_manager.gd`

### 决策 35：王宫为可选建造的特殊建筑

**选定方案**：
- 王宫为特殊建筑（category: "special"），每国限建 1 座
- 非首都默认建筑，需要玩家主动花费 400金+100铁 建造
- 3 级可升级，提供民心+税收+外交声望的综合加成

**理由**：
- 可选建造增加玩家决策点：是先建王宫还是先建经济/军事建筑
- 高成本体现"营建宫室"的重量感
- 综合加成（民心+5/级、税收+10%/级、声望+5/级）使其成为中期战略目标
- 阶段3+增加首都前置条件
**已落地**：`data/buildings.json`

### 决策 36：历史事件选项标记采用 `(历史)` 前缀

**选定方案**：
- 历史选项的 `text` 字段中加 `(历史)` 前缀，如 `"text": "(历史) 减灶诱敌，伏击马陵道"`

- 策划可快速筛选历史选项进行内容审查

### 决策 38：历史事件触发条件采用三类机制

**选定方案**：
- **类型A（战争事件）**：需要 `at_war_with`（交战敌国ID）+ `troops_in_hex_range`（六角坐标范围判定双方军队存在）
- **类型B（建造事件）**：通过 `wonder_built` / `building_built` 条件触发（本批事件未使用，预留扩展）
- **类型C（回合事件）**：通过 `turn_min`/`turn_max` + `faction`/`season`/`morale` 条件触发
- 战争事件的中心坐标暂用占位值，后续根据地图实际坐标调整

**理由**：
- 三类机制覆盖战国事件的主要触发场景：战争、建设、时序
- 类型A的地理限定使战争事件更贴合历史（长平→上党附近，马陵→大梁附近）
- 类型C的国家限定通过 `conditions.faction` 实现，概率0.05~0.12
- 预留 `prerequisite_event` 字段支持事件依赖链（阶段4扩展）
**已落地**：`data/events.json`、`scripts/autoload/event_manager.gd`（需扩展条件检查）

### 决策 39：枪刺兵定位为反骑兵/反战车专业兵种

**选定方案**：
- 枪刺兵（id: `spear`）：攻击11、防御7、HP90、速度3、射程1
- 特殊标签 `anti_cavalry`，用于战斗系统识别
- 克制：骑兵×1.35、战车×1.30
- 被克：步兵×1.15（轻克）、弓兵×1.25（中克）
- 对弩兵×1.2、攻城器械×0.7、水军×1.0、自相×1.0

**理由**：
- 枪阵（如马其顿方阵、秦枪阵）是克制骑兵冲击的经典战术
- 攻击力介于步兵(10)和骑兵(12)之间，防御略低于步兵，体现"进攻型反骑"定位
- 被步兵轻克体现近身肉搏劣势，被弓兵中克体现远程打击弱点
- 造价60金+15铁，略高于步兵，性价比适中
**已落地**：`data/units.json`、`data/balance_params.json` counter_matrix 8×8

### 决策 40：弓兵弩兵射程统一为2

**选定方案**：
- 弓兵 range=2（不变）、弩兵 range=2（从3减到2）

- 韩国劲弩射程4作为国家特色差异化，不受统一调整影响

### 决策 41：水军限定水域移动 + 冬季搁浅机制

**选定方案**：
- 水军只能在 `is_navigable=true` 的地形上移动（当前仅河流）
- 水军特殊标签从 `amphibious` 改为 `naval`，明确水战定位
- 冬季河流结冰（`terrain.frozen_season: "winter"`），水军进入搁浅状态
- 搁浅状态：不能移动、不能主动进攻、可被陆军攻击
- 搁浅参数通过 `balance_params.json` 的 `naval_freeze` 配置

**理由**：
- 水军应受水域限制，避免"水军上岸"的不合理情况
- 冬季结冰搁浅增加季节策略深度，水军需冬季前撤回安全水域
- `frozen_season` 字段设计为可扩展，未来可支持其他冻结地形
**已落地**：`data/units.json`、`data/terrain.json`、`data/balance_params.json`

### 决策 42：水战攻击规则为双向50%减弱

**选定方案**：
- 水军 vs 陆军（双方在各自地形）：双方攻击力均×0.5
- 陆军在水面 vs 水军：陆军攻击力×0.5、防御力×0.5；水军无额外debuff
- 参数通过 `balance_params.json` 的 `naval_combat` 配置

**理由**：
- 水战双方都不在最优环境，攻击力减半体现适应性差异
- 陆军在水面额外防御debuff体现不习水战的劣势
- 水军在水面无额外debuff，体现水域作战优势
- 配置化设计便于后续根据测试调整水战强度
**已落地**：`data/balance_params.json` naval_combat

### 决策 43：城市扩展至50座 + 地图扩大到30×20

**选定方案**：
- 城市从14座扩展到50座，按历史面积比例分配：楚9/秦8/齐7/赵7/魏6/韩5/燕5/中立3
- 地图从20×15（300格）扩大到30×20（600格），密度从1城/21格调整为1城/12格
- 重点选择经济实力强（如蜀郡、南阳、定陶）或有著名历史事件（如晋阳之战、长平之战、田单火牛阵）的城市
- 中立城市（洛邑、邢台、定陶）使用 `faction_id: "neutral"`，不归属任何国家

**理由**：
- 14城对4X游戏过少，战略纵深不足，50城提供足够的扩张空间
- 按面积比例分配体现国力差异：楚国最大（9城），韩国最小（5城）
- 地图扩大到600格后，50城密度适中（1城/12格），辖区覆盖58%地图
- 数据驱动架构保证新增城市无需修改代码，仅扩展JSON数据
**已落地**：`data/cities.json`

### 决策 44：城市辖区半径从3缩小到1

**选定方案**：
- `jurisdiction_radius` 从 3（覆盖约19格）缩小到 1（覆盖7格）
- 文化扩散距离衰减从3级（1.0/0.5/0.25）改为2级（1.0/0.3）
- 所有50城统一使用 radius=1

**理由**：
- 50城 × 7格/城 = 350格覆盖，占600格地图的58%，合理
- 辖区缩小后城市间边界更清晰，领土争端更频繁，增加策略深度
- 文化扩散衰减加快，文化胜利难度提升，需要更多文化建筑支撑
- 辖区精细化管理，每座城市的建筑和资源决策更有分量
**已落地**：`data/cities.json`、`data/balance_params.json` culture.distance_decay、`docs/数值公式.md` §4.1
