```markdown
# 《山河策》Claude Code 完整配置流程

以下是将《山河策》项目完全接入 Claude Code 的**逐步骤配置流程**。按顺序执行，约 30 分钟即可让 Claude 带着游戏世界知识、技术规范和自动化质量门开始工作。

---

## 第 1 步：项目目录准备（5 分钟）

确保项目根目录已有 `data/`、`scripts/`、`assets/` 等基础目录（参考技术栈文档的目录结构），并在根目录初始化 Git（如未初始化）。

```bash
cd /path/to/shanhece
git init
```

创建 Claude Code 所需的隐藏配置目录：

```bash
mkdir -p .claude/skills
mkdir -p .claude/commands
mkdir -p .githooks
```

---

## 第 2 步：安装 Claude Code 并初始化项目

若电脑尚未安装 Claude Code，请先在终端执行（需 Node.js ≥ 18）：

```bash
npm install -g @anthropic-ai/claude-code
```

进入项目根目录，启动 Claude Code：

```bash
cd /path/to/shanhece
claude
```

首次启动会引导你登录 Anthropic 账号，并在项目根目录自动生成 `.claude/settings.json`。

---

## 第 3 步：配置 MCP 服务器（引擎 + 知识 + 文件 + 可选搜索）

编辑 `.claude/mcp.json`（如无则创建），写入以下内容。每个 MCP 说明见注释。

```json
{
  "mcpServers": {
    "godot-mcp": {
      "command": "npx",
      "args": ["-y", "godot-mcp-server"],
      "description": "连接 Godot 编辑器，用于场景操作、运行项目、调试输出"
    },
    "gamecodex": {
      "command": "npx",
      "args": ["-y", "gamecodex"],
      "description": "游戏设计模式、AI 行为树、4X 常见算法参考（950+ 文档，覆盖 29 引擎）"
    },
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "<YOUR_PROJECT_ABSOLUTE_PATH>"],
      "description": "批量读写 JSON、搜索脚本、校验资产命名"
    }
  }
}
```

> **重要：**
> - `filesystem` 的 `<YOUR_PROJECT_ABSOLUTE_PATH>` 必须替换为你的项目**绝对路径**（如 `/home/user/shanhece`）。
> - MCP 包名已验证：`godot-mcp-server`（v0.5.0）、`gamecodex`（v1.0.0）、`@modelcontextprotocol/server-filesystem`（v2026.1.14）均为 npm 实际存在的包。
> - 建议把 `brave-search` 先注释掉，因为需要 API key（可阶段 3 再添加）。

**配置完毕后的动作：**
1. 重启 Claude Code 以加载 MCP。
2. 在 Godot 编辑器中，安装并启用 Godot MCP 插件（参考 [godot-mcp-server 文档](https://github.com/tomyud1/godot-mcp)）。
3. 在 Claude Code 中输入 `/mcp` 可验证所有服务器是否在线。

# 《山河策》核心游戏世界知识

## 游戏类型与目标
- 类型：回合制历史策略游戏（4X：探索、扩张、开发、征服）

- **4 回合 = 1 年**（春→夏→秋→冬→春），每年春季首回合触发「年度事件」判定
- 默认游戏时长：30~50 年（120~200 回合）

### 4.2 `godot-guidelines.md`

```markdown
# 山河策 Godot 4.3 架构与编码规范

## 引擎与语言
- 引擎：Godot 4.3+ 标准版
- 主力语言：GDScript（**必须**使用静态类型标注 `var health: int = 100`）
- 备选：C#（仅在性能分析确认瓶颈后使用，如大地图寻路、文化扩散）
- 扩展：GDExtension（C++/Rust）（Phase 5 优化，非初始决策）

## 项目目录结构
```
data/                        # JSON 数据文件（units/terrain/tech_tree/events/buildings/factions/diplomacy）
scenes/                      # Godot 场景文件 (.tscn)
  main/                      # 主场景、菜单
  map/                       # TileMap、摄像机
  units/                     # 单位场景
  cities/                    # 城市场景
  ui/                        # UI 面板场景
    hud/                     # 资源栏、回合指示器
    city_panel/              # 城市管理面板
    diplomacy/               # 外交对话框
    tech_tree/               # 科技树视图
    events/                  # 事件弹窗
scripts/                     # GDScript 脚本
  autoload/                  # 全局单例管理器（game_manager/data_manager/event_manager/signal_bus）
  map/                       # 地图相关脚本
  units/                     # 单位行为
  cities/                    # 城市逻辑
  ai/                        # AI 决策（diplomacy_ai/military_ai/economy_ai）
  systems/                   # 游戏系统（combat_system/culture_system/tech_system/diplomacy_system）
assets/                      # 美术和音频资源
  sprites/                   # 像素美术
  tilesets/                  # TileSet 资源
  audio/bgm/                 # 背景音乐
  audio/sfx/                 # 音效
  fonts/                     # 字体文件
  themes/                    # Godot Theme 资源
```

## 全局单例 (Autoload)
- `GameManager`：回合循环、游戏状态
- `DataManager`：加载 `data/*.json` 并提供数据访问
- `EventManager`：随机事件调度
- `SignalBus`：信号总线，系统间解耦通信

## 信号驱动架构
- 回合结束信号链：GameManager.end_turn() → 发射 turn_ending → 各系统处理 → 发射 turn_started → UI 更新
- 系统间严禁直接调用方法，必须通过 SignalBus 发射信号并连接

## 地图与寻路
- 网格：六角网格，使用轴坐标 (q, r)
- 渲染：TileMapLayer 分层（地形底、覆盖、单位、文化）
- 寻路：AStarGrid2D（内置，支持六角模式），自定义权重函数基于地形消耗
- 补给线检测：BFS/泛洪填充

## UI 技术栈
- 全部使用 Godot Control 节点（PanelContainer, VBoxContainer, ScrollContainer 等）
- 主题：项目级 Theme 资源，统一管理字体、颜色、按钮样式
- 科技树：自定义 Control + `_draw()` 或 GraphEdit

## 数据驱动铁律
- 程序严禁硬编码任何游戏数值、国家 ID、学派 ID
- 所有数值必须通过 `DataManager.get_xxx()` 从 JSON 获取
- 兵种属性字段：id, name, attack, defense, speed, range, special
- 地形字段：id, name, move_cost, effects
- 引擎原生数据（Theme、TileSet 配置、自定义 Resource 子类）使用 `.tres`（Godot Resource）格式

## 版本控制与构建
- Git 托管于 GitHub，二进制资产（PNG、音频、字体）使用 **Git LFS** 管理
- 分支策略：`main`（始终可运行）→ `dev`（开发集成）→ `feature/*`（功能分支）
- 可选 CI/CD：GitHub Actions 自动构建、运行测试、生成可下载构建

## 性能注意
- 文化扩散、AI 计算等 CPU 密集逻辑需使用 `call_deferred` 或分帧处理
- 超过 100 城市时，文化算法优化为仅计算邻城或分批

## 测试
- GUT 单元测试针对：战斗公式、寻路、文化传播、AI 决策、存档读写
```

### 4.3 `art-pipeline.md`

```markdown
# 山河策美术资产管线规范

## 基础规格
- 像素图块：32×32 像素
- 调色板：战国低饱和暖色调，限制 32 色以内
- 纹理过滤：Nearest（像素风清晰）

## 地形图块列表
必须产出的基础图块：平原、森林、山地、河流、沼泽、关隘、渡口、栈道、箭楼
（每种至少 1 个基础变体，允许有 2-3 个不同季节或文化染色版本）

## 文件命名规则
- 地形：`tile_[terrain_type]_[variant].png` 例：`tile_mountain_01.png`
- 单位：`unit_[faction]_[type]_[variant].png` 例：`unit_qin_rushi_01.png`
- UI 元素：`ui_[element_name].png` 例：`ui_button_end_turn.png`
- 禁止使用空格、大写、特殊字符

## 创作工具与流程
- 主要工具：Aseprite（推荐）或 LibreSprite
- 源文件保存为 `.aseprite` 格式，导出为 `.png`
- 精灵表由 Godot 的 SpriteFrames 或 AtlasTexture 处理
- 严禁直接修改导出的 PNG，必须在 Aseprite 中编辑后重新导出

## 文化覆盖层
- 使用半透明颜色混合，通过 TileMap 的第二层（覆盖层）实现
- 不同学派可定义不同颜色（儒家青、法家白、墨家黑、道家黄、兵家红、纵横家紫）

## 字体与 UI 美术
- 标题/书法字体：免费中文字体（如思源宋体）
- 正文/UI 字体：Noto Serif CJK 或思源宋体
- 所有字体文件需包含完整 CJK 字符集
- UI 主题背景采用羊皮纸/竹简质感，低透明度

## 音频管线
- 工具：Audacity（免费）
- 素材来源：CC0 素材库
- 风格：战国风传统乐器（古琴、编钟、鼓等）
- 文件位置：`assets/audio/bgm/`（背景音乐）、`assets/audio/sfx/`（音效）

## TileSet 创建流程
1. 精灵表导入 Godot
2. TileSet 编辑器定义碰撞形状和自定义数据层（地形类型、移动力消耗、特殊标记如关隘/渡口）
3. TileMapLayer 使用

## 验收标准
- 所有资产文件必须符合命名规则
- 所有 32x32 图块边缘清晰，无半透明像素
- 文化染色层与地形底图叠加后不产生视觉冲突
```

## 第 5 步：创建自定义命令（Slash Commands）

### 5.1 `/balance.md`

```markdown
你是《山河策》的数值平衡助手。请执行以下操作：
1. 读取 `data/units.json` 和 `data/factions.json`（如存在）。
2. 基于兵种相克矩阵（步→弓→骑→步，攻城器械独立）模拟 200 场 1v1 战斗，使用当前项目中的战斗公式。
3. 输出每个兵种的胜率分布，标记胜率高于 65% 或低于 35% 的单位。
4. 用表格给出调整建议（建议修改的具体属性值范围）。
```

### 5.2 `/event.md`

```markdown
你是《山河策》的半文言事件生成器。请根据用户提供的上下文（或当前游戏状态）生成 3 个随机事件。
要求：
- 每个事件包含：事件标题（四字文言）、条件（季节/学派/民心等）、描述（半文言，30字以内）、选项 A/B（含资源与民心变化）。
- 符合战国语境，引用《史记》《战国策》典故为佳。
- 遵循数据驱动原则，最终输出对应的 JSON 片段（event_id, condition, description, options）。
```

### 5.3 `/audit.md`

```markdown
对 `scripts/` 目录下所有 .gd 文件执行代码审计，重点查找：

### 6.1 `pre-commit`（硬编码与 JSON 校验）

创建 `.githooks/pre-commit`，内容如下：

> **前提：** 钩子脚本需要 Python 3 可用（`python3 --version` 验证）。

```python
#!/usr/bin/env python3
import os, sys, re, json

# 注意：这些正则模式可能产生误报（如测试代码、数据加载初始化）。可根据项目实际情况调整。
HARDCODE_PATTERNS = [
    r'(attack|defense|speed)\s*=\s*\d+',      # 直接赋值属性
    r'faction\s*==\s*"[a-z]+"',               # 字符串国家比较
    r'DataManager\.get_[a-z]+\(\)\s*[+\-*/]\s*\d+' # 数据 + 魔法数字
]

def check_hardcode(file_path):
    with open(file_path, 'r', encoding='utf-8') as f:
        lines = f.readlines()
    issues = []
    for i, line in enumerate(lines, 1):
        for pat in HARDCODE_PATTERNS:
            if re.search(pat, line):
                issues.append(f"{file_path}:{i}: 疑似硬编码：{line.strip()}")
    return issues

def validate_json(file_path):
    try:
        with open(file_path, 'r', encoding='utf-8') as f:
            data = json.load(f)
        # 检查常见必填字段（以 units 为例）
        if 'units' in file_path:
            for unit in data if isinstance(data, list) else data.get('units', []):
                for field in ['id', 'attack', 'defense']:
                    if field not in unit:
                        return [f"{file_path}: 单位缺少必要字段 '{field}'"]
        # TODO: 阶段 0 末锁定 schema 后，启用对 frozen schema 的合规检查（决策 #1）
        # schema = json.load(open('docs/schemas/frozen_schema.json'))
        # validate_against_schema(data, schema)
        return []
    except Exception as e:
        return [f"{file_path}: JSON 解析失败 - {e}"]

if __name__ == "__main__":
    all_issues = []
    for root, dirs, files in os.walk('.'):
        for file in files:
            path = os.path.join(root, file)
            if file.endswith('.gd'):
                all_issues.extend(check_hardcode(path))
            elif file.endswith('.json') and 'data' in path:
                all_issues.extend(validate_json(path))

    if all_issues:
        print("=== 提交拦截：发现以下问题 ===")
        for issue in all_issues:
            print(issue)
        sys.exit(1)
    print("✔ 硬编码与 JSON 校验通过")
```

### 6.2 `pre-push`（资产命名与文档完整性）

创建 `.githooks/pre-push`：

```python
#!/usr/bin/env python3
import os, sys, re

def check_asset_names():
    issues = []
    pattern = re.compile(r'^(tile|unit|ui)_[a-z]+(_[a-z0-9]+)*\.png$')
    for root, dirs, files in os.walk('assets/sprites'):
        for file in files:
            if file.endswith('.png') and not pattern.match(file):
                issues.append(f"资产命名不规范: {root}/{file}")
    return issues

def check_doc_completeness():
    issues = []
    if os.path.exists('docs/策划案.md'):
        with open('docs/策划案.md', 'r', encoding='utf-8') as f:
            content = f.read()
        if '核心玩法循环' not in content or '七大核心系统' not in content:
            issues.append("策划案.md 不完整：缺少核心章节")
    return issues

if __name__ == "__main__":
    issues = check_asset_names() + check_doc_completeness()
    if issues:
        print("=== 推送拦截：发现以下问题 ===")
        for i in issues:
            print(i)
        sys.exit(1)
    print("✔ 资产规范与文档完整性检查通过")
```
