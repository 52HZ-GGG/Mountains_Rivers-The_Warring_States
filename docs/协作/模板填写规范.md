# 《山河策》协作模板填写规范

> 用途：规范程序、美术、策划在使用 AI Agent 填写协作文档时的输出格式。目标是让不同人、不同 Agent 写出来的内容可直接比较、可直接检查，不靠"看懂了"就能过。

---

## 1. 总原则

1. **只填事实，不填推测**。状态、数量、路径必须来自实际仓库，不能凭记忆写。
2. **路径和文件名必须原样复制**。禁止手动修改大小写、下划线或目录拼法。
3. **状态只允许用固定枚举**。不允许写"差不多""基本完成""大致就绪"。
4. **优先使用表格，不写长段落**。Agent 和人都更容易做 diff 和检查。
5. **每条记录必须能追溯到依据**。依据可以是文件路径、测试结果或数据字段。
6. **不确定就写不确定，不要默认填"通过"或"已完成"**。

---

## 2. 状态枚举

所有模板中的状态字段只能使用以下值：

| 状态 | 含义 |
|---|---|
| `待定稿` | 设计仍未冻结，不可进入实现 |
| `待交付` | 需求已明确，等待资源或设计文件 |
| `待接入` | 文件或设计已交付，程序还没接 |
| `待验收` | 已接入或已交付，但还没最终确认 |
| `已完成` | 代码/资源/数据已确认可用 |
| `暂缓` | 当前阶段不做，标注原因 |

---

## 3. 通用字段说明

### 3.1 依据字段

每一行"状态"旁边都建议写依据。依据格式统一为：

```
文件: <路径>
测试: <测试文件名或函数名>
数据: <data/*.json 中的字段路径>
截图/命令: <可复现方式>
```

示例：

| 事项 | 状态 | 依据 |
|---|---|---|
| 城市建造 | 已完成 | 文件: `scripts/autoload/city_manager.gd`，测试: `tests/unit/test_city_manager.gd` |
| 冻土瓦片 | 占位 | 文件: `scripts/ui/skirmish_tile_textures.gd` 注释"暂用平原" |

### 3.2 路径字段

- 必须使用仓库内相对路径。
- 不要加 file://、vscode://、https://。
- 不要写"同上""见上面"，每一行都要独立。

### 3.3 负责人字段

只填角色或账号，不要写"暂定""TBD""某人"。
如果确实没人接，就写 `未分配`。

## 4. 美术交付模板填写规范

## 5. 策划交付模板填写规范

文件：`docs/协作/策划交付模板.md`

### 5.1 必填字段

| 字段 | 要求 | 错误示例 | 正确示例 |
|---|---|---|---|
| 机制名称 | 具体机制 | "系统优化" | 学派选择 UI |
| 本阶段一定要实现 | 最小闭环 | "全部做好" | 选学派后 buff 生效 |
| 本阶段明确不做 | 边界限定 | "再看" | 学派切换动画 |
| 数据文件 | 仓库路径 | "JSON 里有" | `data/schools.json` |
| 字段名 | JSON 字段原样复制 | "buff 字段" | `level_effects.food_consumption_reduction` |
| 验收标准 | 可检查条件 | "看起来对" | 玩家选择儒家后民心加成生效 |

### 5.2 数值来源规范

所有平衡数值必须写明来源，不允许写"参考设计""适当调整"。

| 数值项 | 来源字段 | 默认值 | 说明 |
|---|---|---|---|
| 粮食季节修正 | `balance_params.json -> resources.season_food_mod` | 按季节查表 | 加法叠加 |
| 税率档位 | `balance_params.json -> tax.rates` | [0, 10, 20, 30, 40] | 5 档 |

### 5.3 边界规则规范

必须覆盖以下场景，不涉及则写"不适用"：

| 场景 | 必须说明 |
|---|---|
| 资源不足 | 选项变灰/不可点击/提示 |
| 重复触发 | 冷却/幂等/报错 |
| 目标不存在 | 降级/跳过/报错 |
| 阶段未解锁 | 灰色/隐藏/提示 |

### 5.4 填写示例

```markdown
# 策划交付：学派选择最小闭环

> 交付日期：2026-05-29
> 交付人：52HZ-GGG
> 对应阶段：阶段 3
> 当前状态：待程序接入

## 1. 本次目标

- 本阶段一定要实现：玩家可选择学派，选完后该学派的 `level_effects` 立即生效
- 本阶段明确不做：学派切换动画、多学派并存、学派技能树
- 对应玩法入口：主菜单 -> 学派选择面板

## 2. 规则说明

- 触发条件：玩家点击学派卡片
- 处理流程：读取 `schools.json` 中对应学派数据 -> 应用基础修正 -> 记录当前学派
- 结算结果：学派 buff 立即生效，UI 显示当前学派
- 边界情况：重复选择同一家不做任何操作

## 3. 数据设计

| 文件 | 字段 | 类型 | 示例 | 说明 |
|---|---|---|---|---|
| `data/schools.json` | `level_effects.food_consumption_reduction` | float | 0.5 | 道家粮耗减免 |
| `data/schools.json` | `level_effects.building_upkeep_reduction` | float | 0.25 | 墨家建筑维护费减免 |

## 4. 数值来源

| 数值项 | 来源字段 | 默认值 | 说明 |
|---|---|---|---|
| 粮耗减免 | `data/schools.json -> level_effects` | 按学派查表 | 加法叠加 |

## 5. UI / 美术需求

| 需求项 | 是否需要 | 说明 |
|---|---|---|
| 新面板 | 是 | 学派选择面板 |
| 新按钮 | 否 | 沿用已有按钮风格 |
| 新插图/图标 | 是 | 6 个学派图标 |
| 新音效 | 否 | 本阶段不做 |

## 6. 验收标准

| 检查项 | 标准 |
|---|---|
| 逻辑 | 选学派后 buff 立即生效 |
| 数据 | 数值从 `schools.json` 读取，无硬编码 |
| UI | 当前学派高亮显示 |
| 测试 | 无当前测试，本阶段不新增 |

## 7. 依赖关系

| 依赖项 | 负责人 | 未完成时如何处理 |
|---|---|---|
| 学派图标 | 美术 | 用临时色块占位 |
| 数据校验 | 程序 | DataManager 已有数据加载 |
```

## 8. Agent 使用时的检查流程

AI Agent 在填写以上任何模板时，必须按以下顺序检查：

1. **先读仓库，不靠记忆**
   - 确认文件是否存在：`Get-Item` / `Test-Path`
   - 确认代码是否引用：`rg` 搜索
   - 确认 JSON 字段是否存在：读 `data/*.json`

2. **再填状态，必须有依据**
   - 每个"已完成"都必须有对应文件或测试
   - 每个"待交付"都必须有对应需求单
   - 每个"待定稿"都必须有未冻结的字段或规则

3. **然后检查路径一致性**
   - 代码引用路径 = 实际文件路径 = 模板中填写路径
   - 三者必须完全一致，不允许"差不多"

4. **最后检查必填字段**
   - 日期、负责人、状态、路径、依据不能留空
   - 不确定的写"未确认"，不要写"不适用"（除非真的不适用）
