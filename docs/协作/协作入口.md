# 《山河策》团队协作入口

> 用途：作为程序、策划、美术协作时的统一入口。本文不替代现有设计文档，而是把交付流程、验收口径、文件入口集中到一处，避免多人进度不同步。

## 1. 当前协作原则

1. **先定需求，再交文件**：策划或程序必须先写清需求、路径、格式和验收标准，美术再按清单交付。
2. **先验收，再接入主线**：文件存在不等于可用，必须通过命名、路径、尺寸、格式、Godot 导入和实际显示检查。
3. **先冻结字段，再写代码**：所有平衡数值和机制字段必须进入 `data/*.json`，字段名只许加不许改。
4. **先区分状态，再推进任务**：每项内容必须标明是“设计中”“待交付”“待接入”“待验收”还是“已完成”。
5. **主分支始终可启动**：任何资源、数据、场景或脚本改动都不能破坏 Godot 编辑器启动。

## 3. 工作状态定义

| 状态 | 含义 | 进入下一状态的条件 |
|---|---|---|
| 待策划定稿 | 有想法，但规则、字段或数值未定 | 策划提供机制说明、字段、边界规则和验收标准 |
| 待美术交付 | 程序或策划已定义资源需求 | 美术按指定路径、命名、尺寸、格式交付文件 |
| 待程序接入 | 文件或数据已交付，但尚未接入代码/场景 | 程序完成加载、显示、逻辑调用或数据读取 |
| 待验收 | 已接入，但还未确认可用 | 通过 Godot 启动、目标场景检查和必要测试 |
| 已完成 | 可运行、可显示、可复现 | 记录到进度文档或交付清单 |
| 暂缓 | 当前阶段不做 | 标明原因和恢复条件 |

## 4. 美术交付规范

### 4.1 交付前必须明确

每个美术需求必须至少包含：

| 字段 | 示例 | 说明 |
|---|---|---|
| 资源名称 | 秦国都城瓦片 | 人能读懂的名称 |
| 用途 | 大地图城市显示 | 用在哪个系统或 UI |
| 输出路径 | `photos/city/tile_city_qin_capital.png` | 必须是项目内目标路径 |
| 文件格式 | PNG | 默认 PNG，特殊情况另行说明 |
| 尺寸 | 1024x1024 或 32x32 | 按用途确定，不确定时先问程序 |
| 背景 | 透明背景 | UI/图标类通常需要透明 |
| 命名规则 | `tile_city_{faction}_{city}.png` | 必须和代码引用一致 |
| 交付性质 | 占位 / 正式 | 阶段 0~1 可用 CC0 占位，禁止下场画终稿 |
| 验收方式 | Godot 中正常显示，无黑边、无拉伸 | 说明如何判断可用 |

### 4.2 命名规则

优先遵循现有规则：

| 类型 | 命名格式 | 示例 |
|---|---|---|
| 地形瓦片 | `tile_{terrain_type}_{variant}.png` | `tile_mountain_01.png` |
| 城市瓦片 | `tile_city_{faction_id}_{city_id}.png` | `tile_city_qin_yongcheng.png` |
| 首都瓦片 | `tile_city_{faction_id}_capital.png` | `tile_city_qin_capital.png` |
| 单位图标 | `unit_{type}.png` 或 `unit_{faction}_{type}.png` | `unit_infantry.png` |
| UI 元素 | `ui_{element_name}.png` | `ui_button_end_turn.png` |
| 特效 | `fx_{effect_name}_{frame}.png` | `fx_fire_01.png` |

禁止使用空格、大写字母、中文文件名和未说明的缩写。

### 4.3 美术验收清单

美术文件进入“待程序接入”前，至少检查：

| 检查项 | 通过标准 |
|---|---|
| 路径 | 文件位于需求单指定目录 |
| 命名 | 文件名符合命名规则，且和代码/数据引用一致 |
| 格式 | PNG 可正常打开，必要时透明背景正确 |
| 尺寸 | 符合需求单尺寸，不靠程序强行拉伸解决 |
| Godot 导入 | 有对应 `.import` 或打开项目后可生成导入文件 |
| 显示效果 | 目标场景中无黑边、错位、模糊、比例异常 |
| 授权 | 占位素材必须来自 CC0 或可商用来源 |

## 5. 策划交付规范

### 5.1 机制交付必须包含

| 字段 | 说明 |
|---|---|
| 机制名称 | 例如“商路系统”“冻土地形效果” |
| 当前阶段目标 | 本阶段只做哪些最小可玩内容 |
| 不做范围 | 明确本阶段暂不实现的内容 |
| 数据文件 | 需要修改哪个 `data/*.json` |
| 字段定义 | 字段名、类型、默认值、示例 |
| 数值来源 | 平衡数值来自哪个 JSON 字段 |
| 交互流程 | 玩家怎么触发、系统怎么结算 |
| 边界规则 | 资源不足、目标不存在、重复触发等情况 |
| 验收标准 | 如何证明机制已完成 |

### 5.2 策划验收清单

机制进入“待程序接入”前，至少检查：

| 检查项 | 通过标准 |
|---|---|
| JSON 字段 | 字段名已确定，且不修改既有冻结字段 |
| 数值来源 | 没有要求程序硬编码平衡数值 |
| 最小闭环 | 有清晰输入、处理、输出 |
| UI 需求 | 如果需要 UI，已说明展示内容和操作 |
| 美术需求 | 如果需要资源，已列出资源清单 |
| 测试口径 | 核心系统说明应覆盖哪些单元测试 |

## 6. 程序接入规范

程序接收策划或美术交付时，按以下顺序处理：

1. 确认交付项是否在当前阶段范围内。
2. 检查路径、命名、JSON 字段和资源引用是否一致。
3. 若缺少验收标准，退回补充，不直接猜实现。
4. 接入时保持数据驱动，平衡数值从 `data/*.json` 读取。
5. 核心系统改动补充或更新 GUT 单元测试。
6. 接入完成后更新进度文档或对应交付清单。

## 10. 遇到不一致时的处理

| 情况 | 处理方式 |
|---|---|
| 文档与代码不一致 | 以当前代码和数据为准，记录文档待更新 |
| 美术路径与代码引用不一致 | 优先修正为需求单指定路径，必要时由程序调整引用 |
| 策划字段与 JSON 现有字段冲突 | 不直接改旧字段，新增字段或开会确认 |
| 资源质量可用但非终稿 | 标记为占位，不阻塞程序接入 |
| 需求超出当前阶段 | 标记暂缓，进入后续阶段池 |
