# 《山河策》程序实现情况说明（策划对接稿）

> **用途**：同步当前工程**已实现 / 未实现 / 缺失与风险**，便于排期、验收与文档预期对齐。  
> **技术前提**：Godot 4.3、GDScript、数值与表结构以 `data/*.json` 为准（禁止硬编码平衡数）。

---

## 一、当前已实现（可运行、有代码支撑）

### 1. 数据与启动

- 启动时由 `DataManager` 加载多份 JSON（地形、兵种、城市、建筑、事件、平衡参数、奇观、势力、外交、科技树、战术演武配置等），并做基础校验；访问接口见 `docs/接口文档.md` 及实际扩展。
- 项目可在编辑器中运行；主场景接入外交 UI、科技 UI、战术演武入口。

### 2. 全局战略回合框架

- `GameManager`：阶段状态机（开局 → 回合开始 → 行动中 → 回合结束）、多势力轮流、回合数递增。
- 全局信号：`turn_started` / `turn_ended` / `phase_changed` / `game_over` 等（详见 `SignalBus`）。
- 玩家侧简化资源池：粮 / 金 / 铁 / 民心 / 人口 / 兵力及增减接口；AI 各国另有资源字典；难度相关读取配置。
- **胜利判定（战略层）**：与灭国/存活逻辑配合（依赖城市管理里的势力与城市状态），用于「仅剩一国」类征服胜负。

### 3. 城市管理（大地图数据侧）

- 多城运行时状态；建造 / 升级 / 拆除 / 建造队列；全国限建等规则按策划决策文档实现。
- 占领易主、迁都、灭国判定等与 `GameManager.check_victory` 衔接的数据与逻辑路径。

### 4. 事件系统（阶段一文档约定的基础形态）

- 回合开始时对**玩家**侧事件判定（时机、条件、概率、冷却）。
- 支持无选项直接结算与有选项事件（通过信号交给 UI，再由接口回填）。
- **说明**：复杂 DSL、战斗中触发、`troops_in_hex_range` 等扩展条件，文档里多列为后续阶段或待扩展。

### 5. 外交与 AI（阶段二量级）

- 好感、声望、条约、战争状态、附庸、商路、通行权、回合衰减与结算等。
- `DiplomacyAI`：与好感/地缘等结合的简化决策辅助。
- **说明**：与策划案「阶段一不写外交」存在文档口径差异——代码里已有外交框架，可作为可选模块或单独验收项。

### 6. 科技系统（阶段二量级）

- 科技前置、研究进度、完成效果汇总（多类修正器）；玩家按回合推进研究。
- AI 侧科技研究为**简化实现**（偏即时完成逻辑），后续可改为与玩家对称的回合消耗。
- **特殊条件**：部分与城市控制、建筑、辖区相关的判定仍为占位或简化对接。

### 7. 战术演武 MVP（阶段一「地盘争夺」方向的独立入口）

- 独立小棋盘（当前配置为 **7×7** 六角轴向格），地形与初始兵种来自 JSON。
- **移动**：按地形移动力消耗寻路，阻挡友军占格；河流 `-1` 移耗等地形规则表现为不可通行。
- **战斗**：单次结算伤害 —— 使用兵种攻防、防守格地形、民心对攻击修正（攻击方）、随机波动、`balance_params` 中克制矩阵与伏击等参数。
- **胜负**：单位进入对方「城格」即判战术胜利；敌方 AI：**随机可行走步 + 射程内随机攻击**。
- **单元测试**：六角工具、战斗结算、战术管理器有关键路径测试。

## 二、当前未实现或仅为占位（与策划案/总案对照）

### 1. 大地图「完整阶段一单线」

- 策划案阶段一表述含：**简易网格地图上的出征单位长期存在、在大地图上随机移动/攻击、一城对一城地盘争夺主循环**。
- **现状**：战术 playable 内容主要在 **独立战术演武场景 + JSON 配置**，尚未把「全国 `cities.json` 六角坐标」与「战场格子」做成**同一套大地图战略 + 切入战斗**的闭环。

### 2. 战斗与战术扩展（文档中有、代码未全覆盖）

- 夹击、火攻、断粮（图论连通）、关隘特殊规则、水军冬季搁浅等：**多数仅在数值文档或 JSON 预留**，战术 MVP **未完整落地**。
- 远程命中率修正（`ranged_accuracy_mod`）等：**结算侧若未接入则等同于未实现玩法差异**。

### 3. 学派与文化

- `DataManager` 中学派接口仍为占位；文化传播、胜利条件等与大地图联动 **未实现**。

### 4. 存档与发布

- 策划案阶段五中的存档/读档、引导、性能优化等：**未作为完整系统交付**。

### 5. 美术管线上的「正式地图表现」

- 战术层当前以 **UI 格子按钮占位**为主，非 TileMap 图层叠地形立绘；不影响逻辑自洽，但和「10×10 网格美术交付」的验收需单独对齐。

## 三、建议与策划重点对齐的「缺失项清单」

以下内容建议在评审会上**逐项确认口径**，避免验收争议：

| 主题 | 建议对齐问题 |
|------|----------------|
| **阶段一验收边界** | 是否以「战术演武 JSON + 占敌城格胜利」为阶段一程序交付终点，还是以「必须从大地图城市入口进入同一战斗规则」为准？ |
| **文档冲突** | 策划案写阶段一「暂不外交/文化」，工程已实现外交与科技——是否归类为「提前迭代」还是「阶段一 scope 需修订」？ |
| **地图尺度** | 策划写 5×5 / 10×10，当前战术配置为 7×7 —— 是否接受或需改为约定尺寸？ |
| **事件条件** | 历史事件里战争地理、`troops_in_hex_range` 等 —— 阶段二再实现还是战术层优先补一条简化判定？ |
| **AI 深度** | 战术 AI 当前为随机策略；是否需要「最低限度」的靠近主城/保护主城等规则才算阶段一完成？ |

## 四、一句话摘要（给策划快速扫）

**已实现**：数据管线、战略回合骨架、城市管理与占领链、事件基础、外交与科技（偏阶段二）、独立战术演武（移动 + 战斗结算 + 占城胜）及对应单元测试。

**未闭合**：大地图与战术战场的统一入口与同一套「从一城打到另一城」的主循环；战斗扩展机制大部未接；学派/文化/存档未做。

**对接关键**：先锁定 **阶段一的程序验收定义**（战术 MVP 即可 vs 必须大地图串联），再排下一迭代优先级。
