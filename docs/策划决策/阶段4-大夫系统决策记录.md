## 系统定位

**大夫系统完全替代旧将军系统**（决策 #2）。旧系统的武力/智力/统率三维属性、5级经验、5技能槽框架全部废弃。大夫是独立于部队和建筑的角色系统，覆盖城市管理、军队统帅、外交使节三大职能。

## 关键决策

### 决策 101：大夫分类与容量

**选定方案**：
- 三类大夫，各有独立容量上限：

| 类型 | 容量上限 | 任命位置 | 核心功能 |
|:---|:---:|:---|:---|
| 文大夫 | 5 | 城市 | 增加城市营收 + 稳定度 |
| 武大夫 | 5 | 部队 | 提供攻防 buff + 特殊技能 |
| 外交大夫 | 2 | 指定目标国家 | 降低外交成本 + 被动增长友好度 |

**理由**：
- 三类大夫覆盖游戏三大维度（内政/军事/外交），形成互补
- 容量上限 5/5/2 限制人才囤积，迫使玩家做取舍
- 外交大夫容量最低（2），体现外交使节的稀缺性和高价值

---

### 决策 102：大夫属性系统

**选定方案**：
- 每类大夫有二维百分比属性，影响其核心功能：

| 类型 | 属性 1 | 属性 2 | 属性 1 效果 | 属性 2 效果 |
|:---|:---|:---|:---|:---|
| 文大夫 | 理财% | 安民% | 城市金钱产出 +N% | 城市稳定度 +N |
| 武大夫 | 勇武% | 韬略% | 部队攻击力 +N% | 部队防御力 +N% |
| 外交大夫 | 辩才% | 亲和% | 外交行动资源消耗 -N% | 目标国友好度 +N/回合 |

**属性公式**：

```
文大夫：
  city_gold_output  *= (1 + 理财% / 100)
  city_stability    += 安民%

武大夫：
  army_attack       *= (1 + 勇武% / 100)
  army_defense      *= (1 + 韬略% / 100)

外交大夫：
  diplomacy_cost    *= (1 - 辩才% / 100)    # 最低降至 50%
  target_opinion    += 亲和% / 10            # 每回合，四舍五入
```

**理由**：
- 二维百分比属性直观、可叠加、易于平衡
- 文大夫同时影响营收和稳定度，体现"治城"的综合性
- 武大夫勇武/韬略替代旧武力/统率，名称更贴合战国语境
- 外交大夫辩才%降低行动成本，亲和%提供被动增长，主动+被动双轨

---

### 决策 103：大夫档次系统

**选定方案**：
- 三档：普通、稀有、传说，核心差异如下：

| 档次 | 属性上限 | 初始技能槽数 | 历史名人 | 概率 |
|:---|:---:|:---:|:---|:---:|
| 普通 | 低（1~30） | 1 | 无 | 60% |
| 稀有 | 中（1~60） | 2 | 可含历史名人 | 30% |
| 传说 | 高（1~100） | 3 | 专属历史名人 | 10% |

**属性范围**（每个属性独立随机）：

| 档次 | 文大夫（理财/安民） | 武大夫（勇武/韬略） | 外交大夫（辩才/亲和） |
|:---|:---:|:---:|:---:|
| 普通 | 5~30 | 5~30 | 5~30 |
| 稀有 | 15~60 | 15~60 | 15~60 |
| 传说 | 30~100 | 30~100 | 30~100 |

**历史名人规则**：
- 传说级：全部为预设历史名人（如白起、商鞅、张仪），拥有专属特殊技能
- 稀有级：部分为历史名人（如赵括、苏代），拥有特殊技能
- 普通级：全部随机生成，无特殊技能
- 随机生成时名字从预设姓氏+名字库中组合

**理由**：
- 三档梯度提供明确的成长感和收集欲
- 传说级属性上限 100 意味着满属性传说大夫极为稀有
- 历史名人与传说/稀有绑定，增强代入感
- 60/30/10 概率池确保普通大夫易得、传说大夫珍贵

---

### 决策 104：大夫技能系统

**选定方案**：
- 技能槽数量由档次决定（普通1/稀有2/传说3），初始即有
- 技能不增加基础属性，仅提供特殊效果
- 技能可升级：每次升级提升对应效果数值
- 特殊技能只有指定历史名人可拥有

**技能框架**：

| 类型 | 技能方向 | 示例效果（框架，具体数值平衡阶段定） |
|:---|:---|:---|
| 武大夫 | 士气联动 | 士气高于 70 时，攻防 buff 额外 +50% |
| 武大夫 | 地形精通 | 特定地形（森林/山地）战斗时攻防 +N% |
| 武大夫 | 追击 | 击败敌军后有概率追击造成额外伤害 |
| 文大夫 | 劝农 | 城市粮食产出额外 +N% |
| 文大夫 | 兴商 | 城市金钱产出额外 +N% |
| 文大夫 | 教化 | 城市稳定度每回合额外恢复 +N |
| 外交大夫 | 巧舌 | 外交行动资源消耗额外 -N% |
| 外交大夫 | 结交 | 目标国友好度每回合额外 +N |
| 外交大夫 | 离间 | 降低目标国与第三国的关系 |
| 通用 | 名望 | 全国士气每回合 +N |
| 通用 | 识人 | 大夫获取概率 +N% |

**特殊技能**（仅历史名人）：
- 白起「人屠」：击败敌军时额外俘获部队
- 商鞅「变法」：在任城市所有产出 +N%
- 张仪「连横」：外交行动成功率 +N%
- 具体名人及技能在平衡阶段设计

**技能升级**：
- 每次升级提升效果数值（如 +5% → +8% → +12%）
- 升级来源：大夫升级时选择一个技能提升
- 技能最高等级：3 级

**理由**：
- 技能聚焦特殊效果而非基础属性，与升级的属性提升形成互补
- 框架先行，具体数值和效果在平衡阶段细化
- 历史名人专属技能增强角色辨识度和收集价值
- 3 级技能上限控制成长天花板

### 决策 105：大夫升级系统

**选定方案**：
- 最高等级：5 级
- 按类型自动获得经验：

| 类型 | 经验来源 | 每回合/每次经验 |
|:---|:---|:---:|
| 文大夫 | 在任每回合 | 20 |
| 武大夫 | 参与战斗 | 30（胜利）/ 15（平局）/ 5（战败） |
| 外交大夫 | 执行外交行动 | 25 |

**升级效果**：
- 每升 1 级，两个属性各 +5%（固定值）
- 每升 1 级，可选择 1 个已有技能提升 1 级
- 5 级满级时属性累计 +20%（4 次升级 × 5%）

**升级经验需求**：

| 等级 | 累计经验需求 |
|:---:|:---:|
| 1→2 | 100 |
| 2→3 | 250 |
| 3→4 | 500 |
| 4→5 | 800 |

**理由**：
- 按类型自动获得经验，与大夫的核心行为绑定
- 文大夫在任即升级，鼓励长期委任
- 武大夫通过战斗升级，鼓励积极用兵
- 外交大夫通过行动升级，鼓励活跃外交
- 每级 +5% 属性提供稳定成长感，5 级累计 +20% 显著但不破坏平衡

---

### 决策 106：大夫获取系统

**选定方案**：
- 四种获取途径：

| 途径 | 说明 | 产出类型 |
|:---|:---|:---|
| 建筑产出 | 学院→文大夫，兵营→武大夫，驿站→外交大夫 | 按建筑固定 |
| 事件触发 | 随机事件中出现招募选项 | 随机 |
| 战斗获得 | 部队击败敌军后概率获得 | 武大夫为主 |
| 科技加成 | 研发特定科技增加获取概率或容量上限 | 不直接产出 |

**建筑产出机制**：
- 建筑每 8 回合产出 1 个大夫（基础速率）
- 产出类型由建筑决定：学院→文大夫，兵营→武大夫，驿站→外交大夫
- 产出时从概率池中抽取档次

**战斗获得机制**：
- 战斗胜利后 15% 概率获得 1 个武大夫
- 大胜（战损比 > 2:1）概率提升至 25%

**概率池**：
- 基础概率：普通 60% / 稀有 30% / 传说 10%
- 可通过科技、建筑、学派调整

**学派影响获取**：

| 学派 | 影响 |
|:---|:---|
| 儒家 | 文大夫获取概率 +20% |
| 法家 | 文大夫获取概率 +10%，武大夫获取概率 +10% |
| 墨家 | 文大夫获取概率 +10% |
| 道家 | 所有大夫获取概率 -10%（道家崇尚无为） |
| 兵家 | 武大夫获取概率 +20% |
| 纵横家 | 外交大夫获取概率 +20% |

**理由**：
- 建筑产出作为稳定来源，确保玩家持续获得大夫
- 战斗获得激励积极用兵，武大夫可通过实战获取
- 事件触发提供惊喜感和历史代入感
- 科技加成提供长期规划维度
- 学派影响获取概率，与现有学派系统联动

### 决策 107：大夫命运机制

**选定方案**：

**武大夫（部队被歼灭时）**：

| 结果 | 概率 |
|:---|:---:|
| 死亡 | 30% |
| 被俘 | 40% |
| 逃脱 | 30% |

**文大夫（城破时）**：

| 结果 | 概率 |
|:---|:---:|
| 死亡 | 30% |
| 被俘 | 40% |
| 逃脱 | 30% |

**外交大夫**：
- 不因城破/部队被歼触发命运判定
- 固定回合寿命：30 回合（自然死亡）
- 寿命到期后自然死亡，无法避免

**被俘处置**：
- 被俘后可花费金币赎回（费用 = 大夫等级 × 100 金）
- 可通过外交谈判要回（消耗外交行动）
- 若未达成赎回/谈判，被俘大夫有概率：
  - 自杀（40%）：永久失去
  - 归顺对方（60%）：加入对方阵营
- 若对方大夫容量已满，被俘大夫自动死亡

**战斗保护规则**：
- 城市或部队上一回合遭遇进攻时，该位置的大夫不能卸任
- 防止玩家通过临时卸任规避命运判定

**理由**：
- 30/40/30 概率分布平衡了风险和可恢复性
- 被俘可赎回增加了外交博弈维度
- 自杀/归顺的二选一增加了戏剧性
- 容量满自动死亡防止大夫无限流转
- 战斗保护规则防止玩家利用卸任机制规避风险
- 外交大夫的固定寿命体现使节的流动性

---

### 决策 108：大夫转移与卸任

**选定方案**：
- 三类大夫转移均需 3 回合
- 转移期间大夫不产生任何效果
- 转移目标：
  - 文大夫：从 A 城市转移到 B 城市
  - 武大夫：从 A 部队转移到 B 部队
  - 外交大夫：从 A 国转移到 B 国
- 卸任：大夫从当前位置移除，进入待命池，可重新任命
- 卸任无回合消耗，但需要当前位置未处于战斗保护状态

**理由**：
- 3 回合转移成本防止玩家频繁调动大夫
- 转移期间无效果，迫使玩家在调动和保留之间做取舍
- 卸任无成本但受战斗保护限制，平衡灵活性和风险

---

### 决策 109：容量扩充机制

**选定方案**：
- 三种途径可扩充大夫容量上限：

| 途径 | 示例 | 扩充量 |
|:---|:---|:---:|
| 科技 | 「郡县制」→ 文大夫 +1，「军功爵制」→ 武大夫 +1 | 每类最多 +2 |
| 建筑 | 「招贤馆」（新建筑）→ 所有类型 +1 | 全局最多 +1 |
| 事件 | 特殊事件奖励（如「稷下学宫」事件）→ 指定类型 +1 | 每类最多 +1 |

**最大容量**：
- 文大夫：5 + 2（科技）+ 1（建筑）+ 1（事件）= 9
- 武大夫：5 + 2（科技）+ 1（建筑）+ 1（事件）= 9
- 外交大夫：2 + 2（科技）+ 1（建筑）+ 1（事件）= 6

**理由**：
- 多途径扩充提供长期目标感
- 每种途径有上限，防止容量无限膨胀
- 最大容量 9/9/6 为后期游戏提供足够空间

### 决策 111：系统交互设计

**与城市系统**：
- 文大夫任命到城市后，`CityManager` 计算产出时读取大夫的理财%/安民%加成
- 文大夫在任期间，城市稳定度每回合额外恢复（安民% 效果）

**与战斗系统**：
- 武大夫任命到部队后，`CombatResolver` 读取大夫的勇武%/韬略%加成
- 替代旧的 `atk_general_mod` 和 `def_general_mod` 占位符（`combat_resolver.gd:29-30`）
- 战斗胜利后触发大夫获取判定和经验获取

**与外交系统**：
- 外交大夫指定目标国家后，`DiplomacySystem` 读取大夫的辩才%/亲和%加成
- 辩才% 降低外交行动资源消耗（最低降至 50%）
- 亲和% 每回合自动增加目标国友好度

**与科技系统**：
- 特定科技解锁容量扩充（如「郡县制」→ 文大夫 +1）
- 特定科技增加获取概率

**与学派系统**：
- 学派影响大夫获取概率（详见决策 106）

**与事件系统**：
- 随机事件可触发大夫招募（如「商鞅入秦」「张仪入楚」）
- 特殊事件可扩充容量

**理由**：
- 大夫系统作为中间层，连接城市/战斗/外交三大系统
- 通过 SignalBus 解耦，各系统只需监听大夫相关信号
- 不修改现有系统的核心逻辑，仅在计算链中插入大夫加成

### 决策 120：Autoload 单例设计

**选定方案**：
- 新增 `MinisterManager` 作为 Autoload 单例
- 文件：`scripts/autoload/minister_manager.gd`

**核心职责**：
- 管理所有大夫的运行时状态（在任/待命/被俘/已死亡）
- 大夫获取、任命、卸任、转移、升级
- 大夫产出判定（建筑/战斗/事件）
- 大夫命运判定（死亡/被俘/逃脱）
- 计算并暴露各系统的加成数值

**核心数据结构**：

```gdscript
# 大夫状态
enum MinisterState { IDLE, ASSIGNED, TRANSFERRING, CAPTURED, DEAD }

# 大夫运行时数据
var _ministers: Dictionary = {}  # id -> MinisterData

class MinisterData:
    var id: String
    var minister_type: String  # "civil" / "military" / "diplomat"
    var tier: String           # "common" / "rare" / "legendary"
    var name: String
    var is_historical: bool
    var level: int
    var exp: int
    var stats: Dictionary      # {"理财": 25, "安民": 18} 等
    var skills: Array          # [{id, level}]
    var state: MinisterState
    var assigned_to: String    # city_id / army_id / faction_id
    var transfer_remaining: int
    var lifespan_remaining: int  # 仅外交大夫

# 各类型容量
var _capacity: Dictionary = {
    "civil": {"current": 5, "bonus_tech": 0, "bonus_building": 0, "bonus_event": 0},
    "military": {"current": 5, "bonus_tech": 0, "bonus_building": 0, "bonus_event": 0},
    "diplomat": {"current": 2, "bonus_tech": 0, "bonus_building": 0, "bonus_event": 0}
}
```

**新增信号**：

```gdscript
signal minister_recruited(minister_id: String)     # 获得新大夫
signal minister_assigned(minister_id: String, target: String)  # 任命
signal minister_unassigned(minister_id: String)    # 卸任
signal minister_transferred(minister_id: String, from: String, to: String)  # 转移
signal minister_level_up(minister_id: String, new_level: int)  # 升级
signal minister_captured(minister_id: String, captor_faction: String)  # 被俘
signal minister_died(minister_id: String)           # 死亡
signal minister_defected(minister_id: String, new_faction: String)  # 归顺
```

**理由**：
- Autoload 单例符合项目现有架构模式
- 独立管理大夫生命周期，不污染现有系统
- SignalBus 信号解耦，各系统按需监听
- 容量计算合并基础值和各类加成，便于扩展
