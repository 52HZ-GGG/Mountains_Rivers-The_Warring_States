# 地理与天堑系统

> 本文档整合《山河策》所有地理、地形、天堑、视野机制细节，作为程序实现的唯一参考源。
> 所有数值参数从 `data/*.json` 读取，禁止硬编码。

---

## 1. 设计概述

11 种地形（含 2 种海洋），通过移动力消耗、战斗修正、天堑阻断、视野遮蔽四维度影响战略决策。天堑地形（山地/河流/深海）需特定科技解锁通过。关隘为可占领特殊地形，拥有 HP，可易主。沙漠与冻土为极端环境地形，补给消耗增加。

**核心设计原则**：

1. 地形修正采用**加法层**，与其他百分比 buff 相加后统一乘算
2. 天堑通行采用**覆盖模式**：科技解锁后 move_cost 被 crossing_rules 覆盖
3. 视野系统采用**预算消耗制**：视野穿过地形消耗 vision_cost，相邻格保底可见
4. 所有数值从 `terrain.json` / `balance_params.json` 读取

**程序实现状态**：数据层完备，运行时仅战术演武层有基础实现（Dijkstra 寻路 + 地形战斗修正）

## 3. 移动力系统

### 3.1 基础移动消耗

每种地形有固定的 `move_cost` 值，单位进入该格消耗对应移动力。

```
actual_cost = terrain.move_cost
```

- move_cost = -1：不可通行（天堑，未解锁科技时）
- move_cost ≥ 1：正常消耗

**数据来源**：`terrain.json` → `move_cost`

### 3.2 天堑通行规则

天堑地形（山地/河流/深海）默认完全不可通行（move_cost=-1）。通行性由 `is_barrier` 和 `crossing_rules` 共同决定：

| 天堑 | barrier_type | 默认状态 | crossing_rules |
|:---|:---|:---|:---|
| 山地 | mountain | move_cost=-1（完全不可通行） | move_cost_add: 3 |
| 河流 | river | move_cost=-1（完全不可通行） | move_cost_add: 3 |
| 深海 | ocean | move_cost=-1（完全不可通行） | 无 |

**关键规则**：所有天堑地形统一使用 move_cost=-1 表示完全不可通行。科技解锁后，move_cost 被 crossing_rules.move_cost_add 覆盖。

**数据来源**：`terrain.json` → `is_barrier`、`barrier_type`、`crossing_rules`

### 3.3 科技破局（覆盖模式）

研究对应科技后，天堑地形的 move_cost 被 `crossing_rules.move_cost_add` 的值**覆盖**：

```
if tech_system.can_traverse_terrain(terrain_id):
    effective_move_cost = crossing_rules.move_cost_add
else:
    effective_move_cost = terrain.move_cost  // 天堑状态
```

| 科技 | 解锁地形 | 覆盖后 move_cost | 前置科技 |
|:---|:---|:---:|:---|
| 栈道之术（plank_road） | 山地 | 3 | 城防术 |
| 浮桥之制（pontoon_bridge） | 河流 | 2 | 城防术 |

**注意**：栈道科技解锁后 move_cost 保持 3（与原值一致），因为山地本身 move_cost=3。浮桥科技解锁后 move_cost 变为 2（与渡口持平）。

**数据来源**：`tech_tree.json` → `terrain_traversal` 效果，`terrain.json` → `tech_overrides`

### 3.4 关隘降速

关隘地形的 `crossing_rules.move_speed_modifier: 0.5` 表示通过关隘时移动力减半：

```
if terrain.special == "pass":
    effective_move_cost = terrain.move_cost × (1 / crossing_rules.move_speed_modifier)
    // move_cost=2, modifier=0.5 → 实际消耗 = 2 × 2 = 4
```

关隘不阻断行军，但大幅降低通过速度。

**数据来源**：`terrain.json` → pass `crossing_rules.move_speed_modifier`

### 3.5 极端环境移动力

沙漠和冻土的 move_cost=3，高于普通地形（平原=1，森林=2）。同时补给消耗增加 20%（见第 9 节）。

**数据来源**：`terrain.json` → desert/tundra `move_cost`

### 3.6 移动力不足通行规则

当单位剩余移动力 < 目标格子 move_cost 时，仍可进入该格子，但有以下限制：

1. 移动力归零
2. 整个回合仅允许 `movement.insufficient_mp_uses_per_turn` 次"不足通行"（默认 1 次）
3. 进入后无法继续移动或攻击（`movement.insufficient_mp_ends_turn: true`）

**设计意图**：避免单位因差 1 点移动力而无法到达相邻格子的尴尬情况。

**数据来源**：`balance_params.json` → `movement`

## 4. 地形战斗修正

### 4.1 攻击修正

地形的 `atk_mod` 作为百分比 buff 参与加法层计算：

```
atk_buff = 1.0
    + (terrain.atk_mod - 1.0)          // 地形攻击修正偏移量
    + other_buffs...                    // 大夫、派系、士气等
```

- atk_mod > 1.0：地形提供攻击加成
- atk_mod < 1.0：地形降低攻击力（如森林 0.9、山地 0.8）
- atk_mod = 1.0：无修正（平原）

**数据来源**：`terrain.json` → `atk_mod`

### 4.2 防御修正

地形的 `def_mod` 作为百分比 buff 参与加法层计算：

```
def_buff = 1.0
    + (terrain.def_mod - 1.0)          // 地形防御修正偏移量
    + other_buffs...                    // 大夫、派系、城墙等
```

| 地形 | def_mod | 说明 |
|:---|:---:|:---|
| 关隘 | 1.5 | 最高防御加成（+50%） |
| 山地 | 1.2 | 高海拔防御优势 |
| 森林 | 1.1 | 树木掩护 |
| 平原 | 1.0 | 无修正 |
| 冻土 | 1.0 | 无修正 |
| 渡口 | 0.9 | 轻微劣势 |
| 沙漠 | 0.9 | 轻微劣势 |
| 河流 | 0.8 | 水中防御劣势 |
| 浅海 | 0.8 | 水中防御劣势 |
| 沼泽 | 0.7 | 地形恶劣 |
| 深海 | 0.7 | 深海防御劣势 |

**数据来源**：`terrain.json` → `def_mod`

### 4.3 伏击机制

伏击概率 = 基础伏击率 + 地形伏击概率，上限由 `combat.ambush_chance_cap` 控制：

```
ambush_p = clampf(combat.ambush_base_chance + terrain.ambush_chance, 0.0, combat.ambush_chance_cap)
```

| 地形 | ambush_chance | 说明 |
|:---|:---:|:---|
| 森林 | 30% | 最高伏击概率 |
| 关隘 | 15% | 关隘地形利于伏击 |
| 山地 | 20% | 山地伏击 |
| 冻土 | 10% | 低伏击概率 |
| 沼泽 | 10% | 低伏击概率 |
| 沙漠 | 5% | 极低伏击概率 |
| 其他 | 0% | 无伏击 |

伏击触发时，伤害乘以 `combat.ambush_damage_multiplier`（默认 2.0）。

**数据来源**：`terrain.json` → `ambush_chance`，`balance_params.json` → `combat.ambush_base_chance`、`combat.ambush_damage_multiplier`、`combat.ambush_chance_cap`

### 4.4 远程精度修正

地形的 `ranged_accuracy_mod` 影响远程攻击的随机波动范围：

```
spread_lo = base_random_spread_lo + terrain.ranged_accuracy_mod × accuracy_spread_factor
spread_hi = base_random_spread_hi + terrain.ranged_accuracy_mod × accuracy_spread_factor
```

- 正值（如山地 +10）：远程精度提高，波动范围缩小
- 负值（如森林 -20）：远程精度降低，波动范围增大
- 关隘 +15：关隘内远程攻击精度最高

**数据来源**：`terrain.json` → `ranged_accuracy_mod`，`balance_params.json` → `combat.accuracy_spread_factor`

## 6. 关隘系统

### 6.1 关隘 HP 与攻防

关隘拥有独立的 HP 和防御值：

| 参数 | 值 | JSON 路径 |
|:---|:---:|:---|
| 关隘 HP | 500 | `fortification.pass_hp` |
| 关隘防御 | 100 | `fortification.pass_defense` |
| 攻城伤害倍率 | ×3.0 | `fortification.siege_damage_multiplier` |
| 地形防御修正 | 1.5 | `terrain.json` → pass `def_mod` |
| 地形攻击修正 | 0.5 | `terrain.json` → pass `atk_mod` |

**关隘战斗公式**（参见战斗系统.md 第 7.5 节）：

```
// 部队 vs 关隘
siege_dmg = attacker_effective_atk × siege_damage_multiplier
pass_remaining_hp = pass_hp - siege_dmg
```

**数据来源**：`balance_params.json` → `fortification`，`terrain.json` → pass

### 6.2 关隘占领与易主

关隘为 `is_occupiable=true` 的特殊地形，可被占领和易主：

1. 关隘 HP 降为 0 时，占领方变更
2. 占领后关隘 HP 恢复至 `capture_restore_ratio`（30%）
3. 未受攻击时，每回合自然恢复 `natural_recovery_ratio`（5%）

**数据来源**：`terrain.json` → pass `is_occupiable`、`structure_hp`，`balance_params.json` → `city_combat`

### 6.3 关隘文化传播阻断

关隘的 `crossing_rules.blocks_culture_except_owner: true` 表示：

- 关隘占领国的文化可以正常传播通过
- 非占领国的文化被阻断，无法穿过关隘传播

**数据来源**：`terrain.json` → pass `crossing_rules.blocks_culture_except_owner`

| 等级 | 名称 | HP | 攻击力 | 射程 | 防御值 | 建筑槽 |

**数据来源**：`balance_params.json` → `city_levels`

### 7.2 城防战斗公式

部队攻击城市时，伤害按比例分配给驻军和城墙：

```
// 有驻军时
garrison_dmg = total_dmg × damage_split_garrison    // 50% 对驻军
wall_dmg = total_dmg × damage_split_wall             // 50% 对城墙

// 无驻军时
wall_dmg = total_dmg × no_garrison_wall_ratio        // 100% 对城墙

// 攻城器械倍率（仅作用于城墙伤害部分）
wall_dmg ×= siege_damage_multiplier                  // ×3.0
```

**数据来源**：`balance_params.json` → `city_combat`

### 7.3 城墙与箭塔

| 建筑 | 基础 HP | HP/级 | 防御加成/级 | 攻击/级 |
|:---|:---:|:---:|:---:|:---:|
| 城墙（wall） | 200 | +150 | +15% | -- |
| 箭塔（arrow_tower） | 150 | +100 | -- | +8 |

**数据来源**：`buildings.json` → wall、arrow_tower

### 7.4 占领与恢复

- 占领后城市 HP 恢复至 `capture_restore_ratio`（30%）
- 未受攻击时，每回合自然恢复 `natural_recovery_ratio`（5%）
- 自然恢复需要 `natural_recovery_requires_no_attack: true`

**数据来源**：`balance_params.json` → `city_combat`

## 9. 补给与地形

### 9.1 极端环境补给消耗

沙漠和冻土的 `supply_cost_modifier: 1.2` 表示在这些地形上行动时补给消耗增加 20%：

```
actual_supply_cost = base_supply_cost × terrain.supply_cost_modifier
```

| 地形 | supply_cost_modifier | 说明 |
|:---|:---:|:---|
| 沙漠 | 1.2 | +20% 补给消耗 |
| 冻土 | 1.2 | +20% 补给消耗 |
| 其他 | 1.0 | 无额外消耗 |

**数据来源**：`terrain.json` → desert/tundra `supply_cost_modifier`

### 9.2 断粮 BFS 连通性

断粮判定使用 BFS 算法，从部队位置向友方城市探索可通行路径：

```
// 伪代码
func is_supplied(unit_pos, friendly_cities, terrain_map) -> bool:
    visited = set()
    queue = [unit_pos]
    while queue:
        current = queue.pop_front()
        if current in friendly_cities:
            return true  // 找到连通的友方城市
        for neighbor in neighbors(current):
            if neighbor not in visited:
                terrain = terrain_at(neighbor)
                if can_traverse(terrain, unit):  // 检查通行性
                    visited.add(neighbor)
                    queue.append(neighbor)
    return false  // 无连通路径，断粮
```

断粮效果：

| 效果 | 值 | JSON 路径 |
|:---|:---:|:---|
| 士气损失/回合 | -10 | `supply.morale_loss_per_turn` |
| HP 损失/回合 | -5% | `supply.hp_loss_per_turn` |

**数据来源**：`balance_params.json` → `supply`

### 10.3 视野衰减规则

视野从单位位置向外扩展，每穿过一个格子消耗该格的 `vision_cost`：

```
// 视野预算 = 兵种 vision_range + 科技加成 + 季节修正
vision_budget = unit.vision_range + tech_vision_bonus + season_modifier

// 视野扩展（类似 Dijkstra）
for each cell in hex_spiral(unit.position, max_range):
    remaining = vision_budget - total_vision_cost_to_cell
    if remaining >= 0:
        cell is visible  // 视野可达
    else if cell is adjacent (6 neighbors):
        cell is visible  // 相邻保底：始终可见周围 6 格
    else:
        cell is not visible  // 视野不足
```

**关键规则**：
- 视野穿过高海拔地形（山地/关隘，vision_cost=3）时消耗大，容易被阻断
- **相邻保底**：所有部队始终可以看到周围 6 格（相邻格子），无论 vision_cost 是否超出视野预算
- 这与移动力保底机制类似（移动力不足仍可进入相邻格子）

## 11. 寻路算法设计

### 11.1 A* 六角格寻路

大地图使用 A* 算法（Godot 内置 `AStarGrid2D`，`cell_shape = CELL_SHAPE_HEXAGON`）：

```
// 启发函数：六角距离
heuristic(cell_a, cell_b) = hex_distance_axial(cell_a, cell_b)

// 边权重：地形 move_cost（考虑科技解锁）
edge_weight(cell) = effective_move_cost(cell.terrain, tech_system)
```

**数据来源**：`docs/六角网格规范.md` 第 6 节

### 11.2 移动力预算计算

Dijkstra 算法计算给定移动力预算内的可达格子（战术演武已实现）：

```
// 伪代码
func dijkstra_reachable(origin, mp_budget, unit_type):
    reachable = {}
    priority_queue = [(0, origin)]
    while priority_queue:
        cost, cell = pop_min(priority_queue)
        if cost > mp_budget:
            continue
        reachable[cell] = cost
        for neighbor in neighbors(cell):
            move_cost = tile_move_cost(neighbor, unit_type)
            if move_cost < BIG_MOVE:  # 可通行
                new_cost = cost + move_cost
                if new_cost <= mp_budget and new_cost < reachable.get(neighbor, BIG_MOVE):
                    priority_queue.push((new_cost, neighbor))
    return reachable
```

**已实现**：`scripts/autoload/tactical_skirmish_manager.gd` → `_dijkstra_reachable()`

### 11.3 战斗移动规则

部队在战斗中的移动遵循以下规则：

1. **预判可达位置**：根据地图地形，使用 Dijkstra 算法预判部队可以行动到的位置
2. **迷雾中断**：战争迷雾中遇到敌人时立即中断行动（`combat.fog_enemy_stops_movement: true`）
3. **移动后攻击**：部队可以移动后攻击，但必须拥有走到敌军单元格上的移动力
4. **相邻保底**：相邻敌军但敌军单元格行动力消耗过高时，有保底机制（类似移动力不足通行：仍可进入但行动力归零）
5. **攻击后停止**：士兵攻击后默认行动结束，不再拥有移动能力（`combat.attack_ends_turn: true`）

**数据来源**：`balance_params.json` → `combat.attack_ends_turn`、`combat.fog_enemy_stops_movement`

## 12. 区域控制（ZOC）

### 12.1 设计概述

区域控制（Zone of Control）是军事单位对周围六角格施加的控制力，迫使敌方单位在进入控制范围时消耗额外移动力。该机制增加战场态势感，使阵型布局和防线构建成为重要策略维度。

**核心规则**：
- 步兵和骑兵单位产生 ZOC，弓兵和攻城器械不产生
- 城市和关隘需要有驻军才能产生 ZOC
- 水军不受陆地 ZOC 影响，也不产生 ZOC
- ZOC 实时更新：单位移动/死亡后控制范围立即消失

**数据来源**：`balance_params.json` → `zoc`

### 12.2 ZOC 产生源

| 来源类型 | 产生 ZOC | 条件 | 说明 |
|:---|:---:|:---|:---|
| 步兵 | ✓ | 无 | 基础控制单位 |
| 骑兵 | ✓ | 无 | 机动控制单位 |
| 弓兵 | ✗ | -- | 远程单位不产生 ZOC |
| 攻城器械 | ✗ | -- | 攻城单位不产生 ZOC |
| 水军 | ✗ | -- | 水军不产生陆地 ZOC |
| 城市 | ✓ | 需有驻军 | 无驻军时不产生 ZOC |
| 关隘 | ✓ | 需有驻军 | 无驻军时不产生 ZOC |

**数据来源**：`balance_params.json` → `zoc.generates_zoc`

### 12.3 ZOC 受影响单位

|:---|:---:|:---|

| 攻城器械 | ✓ | 虽不产生但受限制 |

### 12.4 ZOC 效果与范围

| 参数 | 值 | JSON 路径 |
|:---|:---:|:---|
| 控制范围 | 1 格（6 邻格） | `zoc.base_range` |
| 额外移动力消耗 | +1 | `zoc.extra_move_cost` |
| 效果叠加 | 否 | `zoc.stacking` |
| 水军免疫陆地 ZOC | 是 | `zoc.navy_immune_to_land_zoc` |

**叠加规则**：多个 ZOC 重叠时，效果不叠加。敌方单位进入任意 ZOC 范围时，仅触发一次 +1 移动力消耗。

**示例**：
```
敌方步兵位于 A 格，骑兵位于 B 格。
A 和 B 的 ZOC 范围在 C 格重叠。
友方单位进入 C 格时，消耗 +1 移动力（非 +2）。
```

**数据来源**：`balance_params.json` → `zoc`

### 12.5 科技扩展

| 科技 | ID | 效果 | 前置 | 费用 |
|:---|:---|:---|:---|:---:|
| 控地之术 | zoc_expansion | ZOC 范围 1→2 格 | 列阵之法 + 铁兵锻造 | 250 金 |

**控地之术**：步兵和骑兵的 ZOC 范围从 1 格扩展至 2 格（19 个六角格）。城市和关隘的 ZOC 范围同步扩展。

### 12.6 寻路中的 ZOC 计算

在 Dijkstra/A* 寻路算法中，ZOC 作为额外移动力消耗加入边权重计算：

```
// 寻路边权重（含 ZOC）
func edge_weight(from, to, unit, enemy_units):
    base_cost = terrain_move_cost(to)
    if is_in_enemy_zoc(to, enemy_units) and not has_zoc_immunity(unit):
        base_cost += zoc.extra_move_cost  // +1
    return base_cost
```

**实时更新**：每次单位移动或被消灭后，重新计算 ZOC 覆盖范围，更新寻路权重。

**数据来源**：`balance_params.json` → `zoc.extra_move_cost`
